## Re-apply /10x-bootstrapper patches after every `10x get`

- **Context**: Any time a Module-1 lesson is re-fetched via `10x get` (m1l3/m1l4/m1l5â€¦). The `.claude/skills/10x-bootstrapper/` files are vendored and overwritten with upstream's shipped versions on every fetch.
- **Problem**: Upstream still ships two bugs this repo patched â€” `audit_commands.js` as a bare `npm audit` string (fails `ENOLOCK` on pnpm) and a `.bootstrap-scaffold` temp-dir name (`create-next-app` rejects dot-prefixed names). A fetch silently reverts both, so a later `/10x-bootstrapper` run would break on this pnpm project.
- **Rule**: After any `10x get`, run `git checkout HEAD -- .claude/skills/10x-bootstrapper/` to re-apply the committed patches, then confirm `git status` shows those files unmodified.
- **Applies to**: all

## Verify against a server you confirmed bound â€” not a stale one

- **Context**: Manual/Playwright verification of a Next.js app via a local server. `next start` renames its process to `next-server`, so `pkill -f "next start"` never matches it; the old server keeps holding port 3000 and every "restart" silently fails with `EADDRINUSE`.
- **Problem**: The browser then hits the **stale build**, so code changes appear to have no effect. During F-01 this produced a confident-but-false conclusion ("Zod Standard-Schema validators don't populate field errors") and a needless workaround + wrong comment â€” all because three "fixes" were tested against an unchanged server. A wrong test is worse than no test: it manufactures false certainty.
- **Rule**: Restart by PID/port, not name: `kill $(lsof -i :3000 -sTCP:LISTEN -t)`. Before trusting any test, confirm the **new** server bound (check the start log for `Ready` AND that the listening PID changed / no `EADDRINUSE`). When a fix "has no effect," suspect the harness before re-diagnosing the code.
- **Applies to**: /10x-implement, /10x-impl-review, any manual/E2E verification

## In Playwright, authenticate a supabase-js client with `signInWithPassword`, not browser-cookie reuse

- **Context**: Any E2E test that needs an **authenticated API client** (not just UI clicks) â€” e.g. seeding rows or asserting RLS isolation directly against PostgREST, where no mutation UI exists yet.
- **Problem**: Two traps that make the obvious approach fail. (1) `@supabase/ssr` stores the session as **HttpOnly, base64-chunked `sb-*-auth-token` cookies** â€” there is no JS-accessible token in the browser to "reuse from the signed-in context," so scraping `page.context().cookies()` is fiddly and coupled to ssr's encoding. (2) The Playwright **spec process loads no env** â€” `playwright.config.ts` only reads `process.env.CI`; the webServer gets `.env.local` because Next loads it natively, but a raw spec does not, and `@/lib/env` is a Next path alias unreachable from the test runner. So `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY` is undefined in the spec.
- **Rule**: Build the per-account client with `createClient(url, anonKey)` + `signInWithPassword(email, password)` using the test's own known credentials â€” it returns the `access_token` directly, no cookie extraction. Make the URL + anon key reachable to the spec by loading `.env.local` in `playwright.config.ts` (add `dotenv`, or inline the deterministic local anon key). Drive sign-up through the real UI for the auth-path coverage, but do **data ops** through the programmatic client.
- **Corollary**: Repo read helpers call `createClient()` (server, `next/headers` cookies) which throws outside a request â€” so any helper a test must call has to accept an **injectable** `SupabaseClient` (defaulting to `createClient()` in app code).
- **Applies to**: /10x-plan, /10x-implement, any Playwright E2E needing authenticated API access

## Verify Postgres constraints/schema via pg_catalog, not information_schema

- **Context**: Any time you probe Postgres schema during implement/review/research â€” confirming a FK's `ON DELETE` action, checking RLS policies, grants, or column definitions (e.g. proving `auth.sessions â†’ auth.users` cascades before relying on it).
- **Problem**: `information_schema`'s multi-view FK joins give false negatives for cross-schema FKs and only surface objects the current role has column privileges on. A constraint query returned **0 rows** for the real `auth.sessions â†’ auth.users ON DELETE CASCADE` that `pg_constraint` then confirmed â€” trusting it would have manufactured false certainty ("no cascade exists") and could have reverted a correct decision.
- **Rule**: When verifying constraints/FKs/schema, query the `pg_*` catalog (`pg_constraint`, `pg_class`, `pg_policy`, `pg_proc`) as the authoritative source â€” not `information_schema`. Treat an `information_schema` "absence" as unproven, not confirmed. (Same family as the verify-against-reality / pick-the-authoritative-probe rule.)
- **Applies to**: /10x-implement, /10x-impl-review, /10x-research, /10x-plan-review

## Local-Supabase E2E sign-up is intermittently flaky (GoTrue DB race) â€” don't gate on it

- **Context**: Running the Playwright suite (`pnpm test:e2e`) against the local Supabase stack. Every spec signs up a fresh user through the UI via the shared `e2e/helpers.ts` `signUp`.
- **Problem**: Single GoTrue container under sustained load over a long serial run. Two faces of the same cause: (1) sign-up intermittently fails to reach `/dashboard` and stays on `/sign-up` with an empty error alert ("Database error finding/saving user"); (2) a _later_ nav (e.g. `createNote` â†’ `toHaveURL(/notes/[id])`) times out because `proxy.ts` revalidates the session via GoTrue `GET /auth/v1/user` on every protected navigation, and those calls slow as the run accumulates (observed `/user` durations climbing 38â†’128ms). Not app code â€” it clears after a container restart. Red herrings: `[auth.rate_limit] email_sent` ("requires auth.email.smtp"; local uses Mailpit + `enable_confirmations = false` â†’ sign-up sends no email); `sign_in_sign_ups = 30/5min` is a real ceiling back-to-back reruns _can_ breach, but a 429 fails the sign-up POST (stuck-on-/sign-up), not a later nav.
- **Rule**: Environmental flake, don't gate on it. **Fixed permanently via `retries: 2` in `playwright.config.ts`** â€” `uniqueEmail()` is called inside each test body, so a retry reruns from the top with a brand-new account, side-stepping the race; `trace: 'on-first-retry'` still captures a real regression. Plus read-only specs (`dashboard.spec.ts`) reuse one shared session (a `setup` project saves `storageState`) to cut sign-up load â€” but mutation / empty-state / isolation specs **must** keep fresh-per-test sign-ups (they assert clean-slate / exact counts / distinct users; sharing a session swaps the flake for a contamination bug). For a stubborn local DB: `supabase db reset` (restarts containers â†’ clears GoTrue state).
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E on the local stack

## E2E must build a FRESH server â€” `reuseExistingServer: true` hijacks a running `next dev`

- **Context**: `pnpm test:e2e` while a `next dev` server is up on port 3000 (the normal case when developing one slice and testing another, or two parallel sessions). `playwright.config.ts` had `reuseExistingServer: !process.env.CI` (= `true` locally) with the webServer on the **same port 3000**.
- **Problem**: Playwright saw a server already on 3000 and **silently reused the `next dev` server** instead of running `pnpm build && pnpm start`. `next dev --turbo`'s on-demand compile + Fast Refresh is exactly the hydration race the prod-build was chosen to avoid: the sign-up form submits as a **native GET** (`/sign-up?email=â€¦&password=â€¦` in the URL) before React attaches its `onSubmit`, so it never POSTs to the Server Action â†’ **every sign-up-dependent spec fails deterministically** (~14/15). Misleading because it looks identical to the GoTrue flake above but is 100% reproducible, and a `supabase db reset` is a **red herring** (correlates only because a dev server happened to come up around the same time). Confirm the DB is innocent in one shot: `curl -X POST $URL/auth/v1/signup â€¦` returns a token â†’ the break is frontend, not the DB.
- **Rule**: E2E **always builds and tests a fresh production server, never a reused one** â€” `reuseExistingServer: false`. To let it coexist with a running `next dev`, isolate the e2e server on **both** axes: a dedicated **port (3100)** and a separate build dir (**`NEXT_DIST_DIR=.next-e2e`**, read by `next.config.ts`) so the e2e `next build` doesn't hit the dev server's `.next` build-lock ("Another next build process is already running"). Never flip `reuseExistingServer` back to `true` for speed â€” that is the bug. Diagnose a suspected hijack with `lsof -i :3000` + walking the parent chain to see `next dev` vs `next start`.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E run while a dev server may be up

## Parallel sessions share one working tree â€” stage by explicit path, never `git add -A`

- **Context**: Two Claude Code sessions working this repo at the same time (a normal mode here â€” e.g. S-08 in one, S-04/S-10/S-11 + a `getCurrentUser` cache refactor in another). They share **one on-disk working tree and one git index** â€” there is no per-session isolation unless a `git worktree` was created.
- **Problem**: The other session's half-finished edits sit dirty in the same tree. A `git add -A` / `git add .` (or any "stage everything" commit ritual) sweeps **their** uncommitted files into **your** commit â€” silently mixing unrelated work, polluting the slice's history, and risking shipping a half-built refactor. Same hazard for shared docs: editing `roadmap.md`/`CLAUDE.md` races the other session's edits â€” the Edit tool throws "file modified since read" mid-flight (observed twice closing S-08), and a blind overwrite would clobber their lines. The global `pnpm lint` also fails spuriously (`ENOENT .next-baseline/cache`) when the other session's build dir vanishes mid-glob.
- **Rule**: Stage **only the change's own paths, by name** (`git add <p1> <p2> â€¦`) â€” the `/10x-implement` ritual already prescribes this; honor it literally, never substitute `-A`/`.`. Detect concurrency early: `git reflog` showing commits you didn't author + a dirty tree at session start = another session is live. For shared docs, re-read immediately before each Edit and retry on "modified since read"; commit the doc edit on its own as soon as it lands so the window is tiny. Lint your slice's files explicitly (`pnpm exec eslint <files>`) to get a clean signal when the global run is poisoned by the other session's build dir. When work is genuinely independent and long-running, prefer a **`git worktree` per session** (as S-05 did) â€” it makes cross-contamination structurally impossible.
- **Applies to**: /10x-implement, /10x-archive, any commit/doc-edit step while a second session may be active

## Measure perf in a production build (`next start`), never in `next dev`

- **Context**: Diagnosing a slow page or deciding whether a perf problem is real â€” e.g. the `/subjects/[id]` document response reading **10.09s** in the browser Network tab, which drove a (premature) "this needs caching + a single-pane rewrite" conclusion.
- **Problem**: `next dev --turbo` is uncompiled and unoptimized â€” it compiles routes on demand and **re-runs server work uncached on every request** (RSC render + Shiki tokenization of every note body), so a heavy page's dev timing is routinely **3â€“10Ã— the production number** and is effectively meaningless for perf decisions. The 10s "emergency" was a dev-mode mirage: a real `pnpm build` + `next start` served the same subject in a fraction of the time (`Ready in 890ms`, sign-in 58ms). Trusting the dev figure would have escalated a UX-led experiment (single-pane subject view) into a fake perf fire and possibly justified a caching slice that wasn't needed yet.
- **Rule**: Before treating any latency as a bug or designing a fix, **reproduce it in a production build**. Coexist with a running `next dev` on :3000 by isolating on **both** axes (same trick as the E2E harness): a separate port (`next start -p 3200`) and a separate build dir (`NEXT_DIST_DIR=.next-prodtest pnpm build`, read by `next.config.ts`) so the build doesn't hit the dev `.next` lock. Read the prod number off the Network tab; only then decide. Same family as the verify-against-reality / authoritative-probe rule â€” a dev timing is "unproven," not "confirmed slow." (Independently, the Shiki highlighter loads **all ~200 bundled grammars** when `langs` is unconstrained â€” a measured **3.3s** one-time boot + ~129MB; constrain `langs` to the curated set + `lazy: true` â†’ ~140ms / ~22MB. That's a real, separate cold-start cost, but it is _not_ what the dev mirage was measuring.)
- **Applies to**: /10x-plan, /10x-implement, /10x-impl-review, /10x-research, any perf diagnosis

## URL-driven multiselect filter: debounce to batch, two-mode selection to avoid desync

- **Context**: A server-side list filter whose selection lives in the URL (`?subjects=a,b`) and re-queries on a Server Component on change (the notes "topic" filter; pattern lifted from `wykonczymy`'s `transfer-filters.tsx` + `FilterMultiSelect`). A client island toggles options; the page re-reads the param and re-runs the query.
- **Problem**: Three traps in the client island. (1) **Per-toggle round-trips** â€” committing the URL on every checkbox toggle fires one server re-query _per click_; picking 4 subjects = 4 re-queries + 4 list flashes. The fix is a **debounce that batches** a popover session into one trailing commit â€” this is a deliberate optimization, documented in the reference (`transfer-filters.tsx:66` "Debounce â€¦ batches rapid clicks to reduce how often we hit the server"), **not** incidental. (2) **Optimistic-state desync** â€” if the island seeds `useState` once from the URL prop, an external URL change (Back/Forward, manual edit) won't resync the checkmarks: `useState` ignores a changed initial value. (3) **Orphan timer** â€” a bare `setTimeout` with no cleanup can fire `router.replace` from an unmounted component.
- **Rule**: Use **two-mode selection**: while the popover is **open**, a local `useState` drives (instant toggles, debounce-batched); while **closed**, it's `null` and selection derives straight from the URL prop (`selected = localSelected ?? urlIds`) â€” so closed state always reflects the URL, killing the desync with no second source of truth. **Reseed** local from the prop on open; **flush** any pending debounce immediately on close; **clear** the timer in a cleanup-only `useEffect` (legit teardown, not the derived-state-in-effect anti-pattern the project rule bans). Wrap the commit in `useTransition` so the list stays interactive during the re-query. Keep the generic multiselect a pure controlled primitive (`components/ui/`, sibling to `combobox.tsx`); put the URL/debounce/two-mode logic in the feature-local filter wrapper. Expose **controlled `open`/`onOpenChange`** on the primitive so the wrapper can hook reseed/flush.
- **Meta-rule (why this lesson exists)**: A review flagged the timer-cleanup + desync gaps, and the first "fix" **deleted the debounce entirely** (swapped it for `useOptimistic` firing per-toggle) â€” trading a real, _documented_ optimization for a simpler-looking component. Wrong move: the reference already solved both gaps _with_ the debounce via the two-mode pattern. When review flags an **incomplete** implementation of a deliberate pattern, **complete the pattern** â€” don't strip the optimization. Before "simplifying" code mirrored from a reference, **read the reference's own comments for the rationale** (`rg -i debounce`), and treat a documented design decision as load-bearing until proven otherwise.
- **Applies to**: /10x-implement, /10x-impl-review, /simplify, any URL-state list filter

## Shiki `fallbackLanguage:'text'` emits `--shiki` vars on the `<pre>` only â€” scope highlight assertions to `span[style*="--shiki"]`

- **Context**: Asserting in E2E whether a markdown code block is syntax-highlighted vs plain â€” e.g. the S-13 guard that an unknown fence (` ```xyzzy `) degrades to plain text under `@shikijs/rehype`'s `lazy:true` + `fallbackLanguage:'text'`.
- **Problem**: A `text`-fallback block still renders as `<pre class="shiki" style="â€¦--shiki-dark-bg:â€¦;--shiki-dark:â€¦">` â€” the dual-theme CSS vars sit on the **`<pre>` element**, while the inner token `<span>`s carry **no** style attr. So a "plain?" assertion written as "no `--shiki` anywhere in the `<pre>`" reads **2** and falsely fails even though the fallback worked. (Verified empirically with `codeToHtml('â€¦', {lang:'text', themes})` before writing the assertion â€” count 2 on the `<pre>`, 0 on spans; a real `python` block puts `--shiki` on ~10 token spans.) Second trap: under `lazy:true` a valid _off-list_ language lazy-loads and highlights, so the fallback path only fires for a genuinely unresolvable token â€” the test must use a **bogus** language id, not a real off-list one.
- **Rule**: Distinguish highlighted from plain via **`pre.shiki span[style*="--shiki"]`** â€” `>3` for highlighted, `=== 0` for the fallback. Put the bogus-fence block in its **own note** so the count isn't contaminated by a highlighted block on the same page. Same family as verify-against-reality: render the real output and inspect it before asserting on its shape.
- **Applies to**: /10x-implement, /10x-impl-review, any E2E asserting Shiki highlight vs plain-text fallback

## A page in edit mode can render >1 CodeMirror â€” the shared `fillEditor` `.cm-content` locator goes ambiguous

- **Context**: E2E driving an inline-edit page that mounts more than one markdown/CodeMirror editor at once â€” e.g. S-14's `/notes/[id]?edit=note`, where `NoteForm` (the note body editor) renders **above** `MemoryCardsSection`'s always-present add/edit form (its `code_context` editor). The shared `e2e/helpers.ts` `fillEditor` targets a bare `.cm-content`.
- **Problem**: `fillEditor` was written for the single-editor case (`/notes/new` create: body editor only, since the inline-checks array starts empty). On a page with two editors, `page.locator('.cm-content')` resolves to **2 elements** â†’ Playwright strict-mode violation, `locator.click` throws before any text is typed. The failure looks like a harness bug, but it's a true "which editor?" ambiguity the helper can't resolve.
- **Rule**: On any multi-editor page, don't call the bare shared `fillEditor` â€” scope to the intended instance: `page.locator('.cm-content').first()` (DOM order is reliable: the form rendered first in JSX is first), then `click()` + `evaluate(document.execCommand('insertText', â€¦))` inline (same execCommand trick `fillEditor` uses, to dodge CodeMirror's closeBrackets corrupting ` ``` ` fences). Reserve the bare helper for genuinely single-editor pages (create/new). If multi-editor specs proliferate, add a `nth`/scoped variant to the helper rather than copy-pasting.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E on a page rendering multiple CodeMirror editors

## App-layer `z.uuid()` is stricter than Postgres `uuid` â€” validate DB ids with `z.guid()` (shape), not `z.uuid()` (RFC version/variant)

- **Context**: Validating an id (note / memory-card / subject) that originates in the DB and is echoed back through a Server Action â€” e.g. `/review`'s rating button sends `card.id` to `rateMemoryCard`, which parsed it with `memoryCardIdSchema = z.uuid()`. Surfaced as a red "Invalid memory card id" under the card, with every rating click reverting and no row written.
- **Problem**: Postgres `uuid` accepts ANY 128-bit value; Zod 4 `z.uuid()` enforces RFC-4122 (version nibble `1â€“8`, variant `8â€“b`). Deterministic seed ids like `â€¦-0000-0000-â€¦` (version 0) store fine in PG but `z.uuid()` rejects them, so every id-validating mutation fails at the schema and reverts silently. Doubly hidden: (a) Postgres AND TypeScript both consider the id well-formed, so nothing upstream complains; (b) `z.uuid()` **special-cases the nil uuid** (`00000000-0000-0000-0000-000000000000` passes) â€” so a unit test that used nil stayed green while every real version-0 id failed in the app. A prior session "fixed" it by crafting v4-shaped seed ids (commit `b63cf89`) â€” a DATA workaround that left the contract bug live for the next fixture / migration / externally-supplied id.
- **Rule**: For opaque, server-trusted ids that come from the DB, validate **shape only** with `z.guid()` â€” not `z.uuid()`. The DB owns id validity; the app's only job is to reject non-uuid-shaped junk (`z.guid()` still does that). Existence + ownership are enforced downstream by RLS + the action's re-fetch + the RPC's `if not found` â€” never by the version nibble. Reserve `z.uuid()` for ids you generate yourself and genuinely want to assert are RFC-v4. Lock the regression with a test that accepts a **non-nil version-0** id (nil won't catch it â€” see `src/__tests__/notes-schema.test.ts`). General principle (same family as verify-against-reality / authoritative-probe): validate an invariant at the layer that OWNS it; don't re-assert a stricter invariant the data was never required to satisfy.
- **Applies to**: /10x-implement, /10x-impl-review, /10x-plan, any Zod schema validating a DB-originated id

## dnd-kit `useSortable` spreads `role="button"` onto the row â€” `getByRole('listitem')` won't match it in E2E

- **Context**: E2E (Playwright) asserting on a list whose items are made draggable via `@dnd-kit` `useSortable` â€” e.g. the S-06 subject note-reorder list.
- **Problem**: `useSortable`'s `attributes` (spread onto the sortable element so it's keyboard-operable) include `role="button"` and `tabindex`. That overrides the element's implicit/`<li>` semantics, so `page.getByRole('listitem')` finds **0** rows even though they render â€” the test fails on a true accessibility-tree fact, not a bug.
- **Rule**: Don't select dnd-kit sortable rows by `listitem` role. Target a stable `data-testid` / text content instead, or query the role the widget actually exposes (`button`). If `listitem` semantics matter for a11y, render the drag affordance as a **handle** inside the `<li>` (spread `useSortable` attributes on the handle, not the row) â€” which S-15 plans anyway.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E over a dnd-kit list

## E2E-server port must match Supabase `site_url` / email-template links

- **Context**: Any auth flow whose link is built server-side from a Supabase template â€” password recovery, email confirmation â€” exercised by E2E on the isolated e2e server (port 3100), while local Supabase's `site_url` points at the dev port (3000).
- **Problem**: `recovery.html` hardcoded `{{ .SiteURL }}` â†’ the reset link pointed at `:3000`, but the e2e webServer runs on `:3100`, so the Playwright click landed on a dead port and the spec failed (S-06 en route, fixed `97cbf57`). The mismatch is invisible until a template-driven link is actually followed in a test.
- **Rule**: In email templates use **`{{ .RedirectTo }}`** (honors the per-request redirect the client passes) rather than the static `{{ .SiteURL }}`, so the link follows whichever origin initiated the flow (3000 dev, 3100 e2e). When adding any template-link auth flow, add an E2E that actually clicks the link, and confirm the port resolves.
- **Applies to**: /10x-implement, /10x-impl-review, any auth flow with a server-built email link tested in E2E

## After adding/moving/removing an App Router route, run `next typegen` before `pnpm typecheck`

- **Context**: Any slice that adds, moves, or deletes a `page.tsx`/`layout.tsx`/route segment under `src/app/` (Next 16 has typed routes on). S-15 added `/subjects/[id]/read/*`, then moved it up to `/subjects/[id]/*`.
- **Problem**: `pnpm typecheck` is bare `tsc --noEmit`, which reads Next's **generated** route types under `.next/dev/types/` (and `tsconfig` also globs `.next-e2e/types/**`, `.next-prodtest/types/**`). Those are stale until regenerated, so `tsc` fails with confusing errors against routes that are actually correct â€” `Type '"/subjects/[id]/read"' is not assignable to 'LayoutRoutes'`, or `Cannot find module '.../page.js'` for a route you just deleted. Two faces: (1) a NEW route isn't in the generated `LayoutRoutes`/`PageProps` union yet â†’ false "not assignable"; (2) a stale build dir (`.next-prodtest` from the prod-build-server skill, `.next-e2e` from Playwright) still references a DELETED route's `page.js` â†’ false "cannot find module".
- **Rule**: After any route-file add/move/delete, run **`pnpm exec next typegen`** before trusting `pnpm typecheck`. If the error points inside a stale `.next-prodtest`/`.next-e2e`, `rm -rf` that dir and re-run (both are rebuilt fresh on next use). Same family as verify-against-reality: a stale generated type is "unproven," not "broken code."
- **Applies to**: /10x-implement, /10x-impl-review, any slice touching `src/app/` routes

## App Router bracket segments (`[id]`, `[noteId]`) are git pathspec globs â€” stage with `:(literal)` or a parent dir

- **Context**: Any `git mv`/`git add`/`git rm` on files under bracketed route dirs â€” moving `subjects/[id]/read/*` up to `subjects/[id]/*`, staging a new `[noteId]/page.tsx`, etc.
- **Problem**: Git treats `[...]` in a pathspec as a **glob character class**, so `git add "src/app/(protected)/subjects/[id]/page.tsx"` matches a single-char dir named `i` or `d` â€” i.e. nothing â€” and the command either silently stages nothing or aborts the whole batch with `did not match any files` (one bad pathspec kills the rest, leaving everything unstaged). `git mv` mis-resolves the same way. Quoting in the shell does NOT help â€” it stops _bash_ globbing, but git applies its OWN pathspec glob after.
- **Rule**: For bracketed paths, either (a) prefix the pathspec with the literal magic `:(literal)` (e.g. `git add -- ':(literal)src/app/(protected)/subjects/[id]/page.tsx'`), or (b) stage a non-bracketed **parent directory** with `git add -A -- "src/app/(protected)/subjects"` when the whole subtree is yours (verify nothing foreign is dirty there first â€” parallel sessions share the tree). For deletions, `git add -A -- <path>` (plain `git add <deleted-path>` without `-A` can error). Prefer filesystem `mv` + `git add -A <parent>` over `git mv` for multi-file bracket moves.
- **Applies to**: /10x-implement, /10x-archive, any git file-op on App Router dynamic routes

## E2E selectors: `data-testid` for targeting, role/copy assertions only where the copy IS the contract

- **Context**: Choosing a Playwright locator for any element a spec interacts with. The suite leaned on `getByRole('button', { name: 'Delete' })`, `getByText('â€¦')`, and `getByLabel('â€¦')` â€” locators bound to user-visible copy + a11y wiring. Two recurring pains: (1) renaming a label / aria-text breaks specs that aren't even _about_ the copy (noise, not signal); (2) repeated elements (list cards, the dual desktop+mobile sidebar) match >1 node â†’ strict-mode throws, patched with brittle `.first()` that silently grabs the wrong (hidden) node the day DOM order flips.
- **Problem**: Every selector couples to _something_; the failure mode is coupling to a thing that churns for reasons unrelated to the test. Copy churns constantly (rewording, i18n, tone); DOM structure churns on restyle; an `id` must be page-unique (useless for list rows) and gets co-opted for CSS/anchors. Coupling a locator to copy taxes _every_ refactor that touches wording, even when the test only needed "the submit button." For this solo MVP the team explicitly does **not** want specs failing on label changes.

## Paginated PostgREST reads must handle the out-of-range 416 â€” `.range()` past the last row throws, and supabase-js drops the count

- **Context**: Any `{ rows, total }` list read built on `.select(cols, { count: 'exact' }).range(offset, offset + limit - 1)` (the getDueQueue precedent) â€” notes/memory-cards/subjects list queries. A deep-linked or stale `?page=N` whose `offset` lands past the last matching row.
- **Problem**: PostgREST returns **HTTP 416 / `PGRST103` "Requested range not satisfiable"** when the range start exceeds the row count â€” an **error**, not an empty result. So a query that treats any `error` as fatal (`if (error) throw`) **500s the whole page** on an out-of-range `?page=`. Worse, the Content-Range header still carries the real total (`*/131`) but **supabase-js does NOT parse `count` on the error path** (verified empirically: `count === null` on the 416), so you can't recover the total from the failed call to render a footer. The plan had _assumed_ out-of-range would yield "empty rows + a working footer" â€” it doesn't; it throws.
- **Rule**: Centralize paginated reads in a helper (`src/lib/supabase/run-paginated-query.ts`) that catches `error.code === 'PGRST103'` and falls back to a **lazy head-count** query (`.select(cols, { count: 'exact', head: true })` with the SAME filters, no range) to recover the total â€” returning `{ rows: [], total }` so the footer renders and the user can navigate back, instead of a 500. Build the filtered query via a `filtered(head: boolean)` factory so the ranged read and the count-only fallback share identical `.in()`/`.or()` predicates (the inner-join `notes!inner` count is correct by construction). One round-trip in the common in-range case; the second only fires on the rare overflow. Same family as verify-against-reality: probe the real 416 (`curl` + a supabase-js script) before assuming its shape.
- **Applies to**: /10x-implement, /10x-impl-review, /10x-plan, any `{rows,total}` paginated Supabase read

- **Context**: Any bug fix or debugging task, especially in mutation/persistence paths (Server Actions, RPCs, the recall loop). The inverse direction from risk-first test authoring.

## Build-time env validation in Next.js: 3-file split + gate it in the root layout (not `next.config` + jiti)

- **Context**: Any Next.js env handling where `next build` must FAIL on a missing/invalid env var (client OR server) AND server secrets must stay out of the client bundle. Triggered when adding the first server-only var (e.g. SMTP creds) to a project whose `env.ts` is client-reachable.
- **Problem**: Two facts collide. `import 'server-only'` is the only NATIVE source of a **build-time** failure (a runtime Proxy Ã  la t3-env, or a lazy check, only fails at runtime) â€” but `server-only` is Next-internal (aliased only during app compilation), so a module importing it CANNOT be imported by `next.config.ts` (plain Node), and it poisons the whole module for the client so it can't live in the client-reachable `env.ts`. The instinctive fix â€” validate in `next.config.ts` â€” then drags in `jiti` to load the TS env modules from plain Node, AND forces jiti's **deprecated synchronous** callable: Next loads `next.config` via `require()`, which forbids top-level await, so the modern `await jiti.import()` breaks the build (`require() cannot be used on an ESM graph with top-level await`). That's a whole extra devDep + an unfixable deprecation just to gate env.
- **Rule**: Keep the 3-file split â€” `env-schema.ts` (pure Zod `clientSchema`+`serverSchema`, no side effects, no `server-only`); `env.ts` (client, eager-parse `NEXT_PUBLIC_*` via STATIC `process.env.X` literals so Next inlines them); `env.server.ts` (`import 'server-only'` + eager parse â€” the leak guard). The env layer's internal imports use sibling relative paths (alias-free, so any non-app loader can read them). **Gate the build from the root layout**, not `next.config`: add side-effect imports `import '@/lib/env'` + `import '@/lib/env.server'` at the top of `src/app/layout.tsx` (a server component â€” `server-only` is fine there). Next evaluates the root layout for every route during `next build`, so the eager parse throws at build on a missing/invalid var â€” **no jiti, no `next.config` gate**. Verified both ways: valid env builds clean; `NEXT_PUBLIC_SUPABASE_URL=not-a-url pnpm build` fails with a ZodError during page-data collection.
- **Trade-off** (so this can be reversed knowingly): the `next.config` gate runs unconditionally _by contract_, before compilation; the layout gate depends on Next evaluating route modules at build (true today â€” it imports/renders every route, even dynamic ones, to probe them). Chose the layout gate to drop the jiti dependency. If you ever need the absolute contract guarantee back, the `next.config` + jiti gate is the alternative â€” but use jiti's **sync** callable, never `jiti.import()` (top-level await breaks the require()-based config load). Reference impl: `src/lib/env*.ts` + `src/app/layout.tsx`.
- **Applies to**: plan, implement, research

## Read env vars only through the validated env layer, never raw `process.env`

- **Context**: Any app code (`src/**`) reading env/config/secrets.
- **Problem**: Raw `process.env.X` skips the build-time schema gate (`env-schema.ts` via `next.config.ts`), so a missing var ships silently â€” caused the `OPENROUTER_ENC_KEY` prod outage and the `SUPABASE_JWT_SECRET` token-API 500.
- **Rule**: Read env only via `src/lib/env.ts` (public) / `env.server.ts` (server) â€” never raw `process.env` (`NODE_ENV` exempt). A prod-required var must have no `.default()` (defaults aren't build-gated). Enforced by `no-restricted-syntax` in `eslint.config.mjs`.
- **Applies to**: implement, impl-review, plan-review

## Mock `@/lib/env` in unit tests that import env-touching server actions

- **Context**: Any vitest unit test that imports (directly or transitively) a server action which reads `@/lib/env` / `@/lib/env.server` â€” e.g. importing `sign-up.ts` because it now uses `SITE_URL`. Also triggered the moment you ADD an env import to a server action that tests already import.
- **Problem**: `env.ts` eager-parses `process.env` with Zod at module load. The husky pre-push hook runs `tsc` + `vitest run` in a bare shell where `.env.local` is NOT exported (only Next/vitest's own loaders read it, not the subshell), so the parse throws `ZodError: NEXT_PUBLIC_* expected string, received undefined` before any test body runs â€” failing the test and blocking the push. Hit twice in one session when `sign-up.ts` gained a `SITE_URL` import and its two importing tests weren't updated.
- **Rule**: A unit test importing an env-touching server action MUST `vi.mock('@/lib/env', () => ({ /* only the consts used */ }))` so the module-load parse never runs (mirror `sign-up-email-redirect.test.ts`). And when you add an env import to a server action, run `tsc --noEmit` + the unit suite before committing â€” the pre-push hook runs them envless, so a missing mock fails there, not locally-with-env. Sibling Playwright variant above: "spec process loads no env".
- **Applies to**: implement, impl-review

## Cache-bust on mutation belongs in the Server Action â€” a route-handler `revalidatePath` can't reach the browser's client cache

- **Context**: `nav-cache-staletimes` turned the client Router Cache on (`staleTimes.dynamic = 300`) so repeat nav is instant, then needed every mutation to bust it. Mutations run through BOTH a UI Server Action and a token-API **route handler** (sharing `*-core` modules). The plan put `revalidatePath('/', 'layout')` at every surface â€” including the 6 token-API route handlers, the shared `deleteRowResponse`, and the OpenRouter OAuth callback.
- **Problem**: A **Server Action**'s `revalidatePath` evicts the **calling browser's** client Router Cache â€” the action result carries the invalidation signal back through the Router (Next docs fact: "Server Action â†’ refreshes previously-visited pages on next nav â†’ busting reaches this browser"). A **route handler**'s `revalidatePath` does **not**: it returns a plain HTTP response with no Router channel, so it can only MARK server-side caches, never evict a browser's client cache. AND for this **all-dynamic** app (every authed page reads `cookies()` + supabase-js) there is **no** Full Route Cache and **no** Data Cache to mark â€” so the route-handler call resets nothing that exists. It's a complete no-op, the same family as the existing lesson "`revalidatePath` on a dynamic page is a no-op" â€” now on the route-handler side. The token API is hit by external agents anyway (no browser, no client cache), and the OAuth callback is **doubly** moot: it's a route handler **and** the full-page OAuth redirect already clears the entire client cache on return. Caught at the E2E gate: spec "API write â†’ revisit â†’ fresh" had no valid form â€” within the staleTimes window an open tab stays stale by the design's own admission, and a cold visit to a dynamic page is always fresh regardless of the bust, so a deliberate-break check could never make it red. The 7 route-handler busts were removed; only the **28 Server Action busts** are load-bearing.
- **Rule**: Put the mutation cache-bust in the **Server Action**, never in the shared route handler or `*-core`. Do **not** add `revalidatePath` to a token-API route handler expecting cross-actor browser freshness â€” it cannot reach the tab; cross-actor convergence is bounded by the `staleTimes.dynamic` window (accepted limitation), not by the bust. A route-handler `revalidatePath` does something **only** when the route serves a server-cached output (static / ISR / `'use cache'` + a tag); for dynamic per-user pages it's dead code. Before adding any `revalidatePath`/`revalidateTag`, confirm the **specific cache it targets actually exists** (same family as verify-against-reality). If a risk can't be made to fail a deliberate-break E2E, the behavior it "protects" may not exist â€” investigate before writing the test.
- **Applies to**: /10x-implement, /10x-impl-review, /10x-plan, /10x-plan-review, /10x-e2e, /simplify, any mutation cache-busting across a UI-action + route-handler pair

## A perf optimization that makes the UI show stale persisted state is a correctness bug, not an accepted tradeoff

- **Context**: Any review/list surface where a mutation (rating a card, editing, status changes) updates the DB but the optimization deliberately SKIPS revalidation to keep the next interaction fast â€” `skipRevalidate`, optimistic advance, client buffers, "eventual consistency every N items", prefetch that a `revalidatePath` would invalidate. Surfaced by `review-perf-instant-next-card` Phase 1: the queue-walk skipped `revalidatePath` so the next card was instant, but `/memory-cards` then showed lying due dates ("Due in 2d" after the rating had pushed it to ~10m) until a hard reload. The whole change was reverted 2026-06-17.
- **Problem**: The screen is the product; "the value is correct in the database" is worthless to the user â€” they clicked a button, the persisted state changed, and the UI kept showing the old value. A list that misreports due dates / counts / status after a write is a **bug**, even when the staleness is documented in a plan and the user "signed off" on eventual consistency. Sign-off on an abstract tradeoff ("list refreshes per batch") does not survive contact with the concrete symptom ("I rated it and the date didn't change") â€” the user reads it as broken, and they're right. It hides because typecheck/lint/build/DB-assertions all stay green (the write landed); only a human looking at the screen catches it, and only if they look at the list mid-session (the exact case the optimization bet they wouldn't).
- **Rule**: Never trade truthful UI for perceived speed. After a mutation, the surfaces showing that data must re-render to the new persisted state â€” default to honest revalidation (`revalidatePath('/', 'layout')` for this all-dynamic app) and only optimize with a **targeted** `revalidatePath('/specific-route')` that keeps the view truthful, never by removing the bust. If a plan proposes skip-revalidate / optimistic / batched-refresh, treat "the list may show stale values between refreshes" as a **defect to design out**, not a footnote to accept â€” and don't let "documented + accepted" relabel a lying screen as a feature. When in doubt, correctness over speed; make it fast only after it's honest.
- **Applies to**: /10x-plan, /10x-plan-review, /10x-implement, /10x-impl-review, /10x-e2e, /simplify, any review/list surface with a mutation in its hot path

## A single `pg_dump` is NOT a Supabase recovery artifact â€” use the official roles + schema + data dump

- **Context**: Any "dump prod" / "seed local from prod" / disaster-recovery task on this Supabase project. Surfaced while building `db:dump:prod` / `db:seed:local`: a one-file `pg_dump "$URL" --no-owner --no-privileges --clean | gzip` was used, then restored into the live local stack. Data counts matched prod, so it _looked_ fine â€” but login broke with **"Database error querying schema."**
- **Problem**: Two compounding errors. (1) **Roles live at the cluster level, not inside a database** â€” a plain `pg_dump` of one DB contains zero roles, so it can never fully rebuild prod (`supabase_auth_admin`, `authenticated`, `anon`, â€¦ are missing). (2) Restoring a **full schema dump** (DDL) into a stack that **already has** the schema/roles, with `--no-owner --no-privileges`, recreates the `auth.*`/`public.*` tables owned by the restoring role (`supabase_admin`) and strips every GRANT. GoTrue connects as `supabase_auth_admin`, which then can't read the `auth` schema â†’ the login error. The data load (COPY/INSERT) is independent and still succeeds, so counts match and it masquerades as a working restore. `--no-owner --no-privileges` is correct only when restoring to a _generic_ target; it is wrong for Supabaseâ†’Supabase, where both sides share the managed roles and ownership must be preserved.
- **Rule**: For backup/recovery/local-seed, use the **official Supabase three-dump method** (verified at supabase.com/docs/guides/platform/migrating-within-supabase/backup-restore), which `scripts/db-push-safe.sh` already implements:
  ```bash
  supabase db dump --db-url "$URL" -f roles.sql  --role-only
  supabase db dump --db-url "$URL" -f schema.sql
  supabase db dump --db-url "$URL" -f data.sql   --use-copy --data-only \
    -x storage.buckets_vectors -x storage.vector_indexes
  ```
  Restore all three, in order, triggers off during the data load:
  ```bash
  psql --single-transaction --variable ON_ERROR_STOP=1 \
    --file roles.sql --file schema.sql \
    --command 'SET session_replication_role = replica' \
    --file data.sql --dbname "$TARGET_URL"
  ```
  **Then resync every sequence â€” the step the official docs omit (TESTED 2026-06-29).** A `--data-only` restore does not reliably advance sequences to match the loaded rows, so the restore succeeds and the data is all present, but the **first write fails**: login dies with `duplicate key value violates unique constraint "refresh_tokens_pkey"` â†’ GoTrue returns _"Database error granting user"_. Run after the restore:
  ```sql
  DO $$ DECLARE r record; mx bigint; BEGIN
    FOR r IN SELECT n.nspname sch, s.relname seq, t.relname tbl, a.attname col
      FROM pg_class s JOIN pg_depend d ON d.objid=s.oid AND d.deptype='a'
      JOIN pg_class t ON t.oid=d.refobjid
      JOIN pg_attribute a ON a.attrelid=t.oid AND a.attnum=d.refobjsubid
      JOIN pg_namespace n ON n.oid=s.relnamespace
      WHERE s.relkind='S' AND n.nspname IN ('auth','storage','public') LOOP
      EXECUTE format('SELECT coalesce(max(%I),0) FROM %I.%I', r.col, r.sch, r.tbl) INTO mx;
      EXECUTE format('SELECT setval(%L, %s, true)', r.sch||'.'||r.seq, GREATEST(mx,1));
    END LOOP; END $$;
  ```
  The **same** three-file artifact serves both jobs â€” full DR (restore into a fresh project) and local seed (restore into a blanked local) â€” so what you seed with is byte-identical to your recovery backup. Restore into an **empty** target (fresh project / reset-and-blanked local), never over a live stack. Never trust a single `pg_dump` (no roles) or `--no-owner --no-privileges` for Supabase recovery. **Never "validate" a recovery by row-count alone â€” only a real browser login proves it.** Verified end-to-end 2026-06-29: dump prod â†’ blank local â†’ restore â†’ resync sequences â†’ actual login renders the dashboard with recovered data. Row counts said "perfect" twice while auth was still broken (first by ownership, then by lagging sequences).
- **Read-only / token-free variant â€” the design we shipped (TESTED 2026-06-29)**: `supabase db dump --linked` is cleanest but needs a **full-access account PAT** (Supabase PATs **can't be scoped read-only**) â€” too much privilege for an unattended scheduled backup. The read-only path is the `backup_ro` DB role (`pg_read_all_data`, `bypassrls`, no write), but it **can't run `supabase db dump`** (CLI issues `SET ROLE postgres`, denied) â€” so use **raw `pg_dump`**, scoped to mimic the official shape: `--schema-only --schema=public` (pipe through `sed '/^CREATE SCHEMA public;$/d'`) for schema; `--data-only --schema=auth --schema=public --schema=storage --exclude-table=storage.buckets_vectors --exclude-table=storage.vector_indexes` for data; a **committed `supabase/roles.sql`** for roles (backup*ro can't dump roles either). Raw-`pg_dump` gotchas that bit during the build: its `-x` means `--no-privileges` (use `--exclude-table=`); `--schema=X` \_without* `--schema-only` includes that schema's DATA (FK-violates on restore); `--schema=public` emits a `CREATE SCHEMA public;` that collides with the restore's blank step (strip it). This is what `scripts/db-dump.sh` + `.github/workflows/db-backup.yml` ship â€” same dumpâ†’blankâ†’restoreâ†’resyncâ†’real-login proof, zero token.
- **Post-restore, two errors are EXPECTED â€” not restore bugs, don't chase them**: after restoring a dump into a different environment (e.g. prod data into local), the dashboard logs (1) `AuthApiError: Invalid Refresh Token â€¦ refresh_token_not_found` (400) and (2) an OpenRouter `decryptSecret` failure â€” `decipher.final()` â†’ "Unsupported state or unable to authenticate data". Both are normal re-auth steps, not data loss. (1) The browser still holds a session cookie from the pre-restore `auth.refresh_tokens`, which the restore replaced wholesale â†’ the cookie's token isn't found. Fix: clear localhost cookies / sign out and log in fresh. (2) Encrypted-at-rest columns (the per-user OpenRouter credential) are AES-GCM ciphertext bound to the `OPENROUTER_ENC_KEY` that wrote them â€” **prod's key, not local's** â€” so GCM's auth-tag check fails closed (by design, `src/lib/crypto/aes-gcm.ts`). Fix: re-enter the OpenRouter key in the app once; it re-encrypts under the local key. Do **not** copy prod's `OPENROUTER_ENC_KEY` to your machine to "fix" it. Corollary for a real DR: secrets encrypted at rest are **not portable** across environments â€” a true recovery treats them as a re-login, never as recoverable rows.
- **Restoring into a HOSTED project differs from local â€” the local script does NOT work as-is (PROVEN 2026-06-29)**: the local drill restores as `supabase_admin` (a real superuser) into a blanked stack. A **hosted** project gives only `postgres`, which is **not** a superuser, and its `auth`/`storage` schemas are owned by `supabase_auth_admin`/`supabase_storage_admin`. Restoring the server-made 3-file backup into a fresh hosted project (ref-throwaway, **session pooler** URL â€” port 5432, never the transaction pooler 6543) hit four privilege walls, each fixed: (1) **`function extensions.moddatetime() does not exist`** â€” a `--schema=public` dump captures triggers but NOT their extensions (those live in the excluded `extensions` schema), and a fresh project doesn't enable `moddatetime` â†’ run `create extension if not exists moddatetime schema extensions;` **before** schema. (2) **`ALTER DEFAULT PRIVILEGES FOR ROLE supabase_admin â€¦ permission denied`** â€” `postgres` can't alter another role's default privileges; **strip** those lines from `schema.sql` (managed boilerplate, already present on a fresh project). (3) **`permission denied for table auth.schema_migrations`** â€” framework migration bookkeeping owned by `supabase_auth_admin`; **skip** the `auth.schema_migrations` + `storage.migrations` COPY blocks (not your data, already populated). (4) the manual sequence-resync DO block **fails** on hosted (the only sequence, `auth.refresh_tokens_id_seq`, is owned by `supabase_auth_admin` â†’ `postgres` can't `setval`) â€” but you **don't need it**: `data.sql`'s own `SELECT setval(...)` already synced it during restore. Also **skip `roles.sql`** entirely (all managed roles pre-exist on a fresh project). Net hosted recipe: enable extensions â†’ restore filtered `schema.sql` (no supabase_admin default-priv lines) â†’ restore filtered `data.sql` (no migration-tracking tables) with `session_replication_role = replica` â†’ NO manual resync, NO roles.sql. **Proven by a real password-grant login** (GoTrue verified the restored bcrypt hash, issued a JWT) **+ an RLS-scoped PostgREST read** (user saw only their own 31 of 84 notes) against the restored throwaway project.
- **Applies to**: /10x-implement, /10x-plan, any db dump/restore/seed/disaster-recovery task on Supabase
