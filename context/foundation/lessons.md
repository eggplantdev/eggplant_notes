## Re-apply /10x-bootstrapper patches after every `10x get`

- **Context**: Any time a Module-1 lesson is re-fetched via `10x get` (m1l3/m1l4/m1l5â€¦). The `.claude/skills/10x-bootstrapper/` files are vendored and overwritten with upstream's shipped versions on every fetch.
- **Problem**: Upstream still ships two bugs this repo patched â€” `audit_commands.js` as a bare `npm audit` string (fails `ENOLOCK` on pnpm) and a `.bootstrap-scaffold` temp-dir name (`create-next-app` rejects dot-prefixed names). A fetch silently reverts both, so a later `/10x-bootstrapper` run would break on this pnpm project.
- **Rule**: After any `10x get`, run `git checkout HEAD -- .claude/skills/10x-bootstrapper/` to re-apply the committed patches, then confirm `git status` shows those files unmodified.
- **Applies to**: all

## Verify against a server you confirmed bound â€” not a stale one

- **Context**: Manual/Playwright verification of a Next.js app via a local server. `next start` renames its process to `next-server`, so `pkill -f "next start"` never matches it; the old server keeps holding port 3000 and every "restart" silently fails with `EADDRINUSE`.
- **Problem**: The browser then hits the **stale build**, so code changes appear to have no effect. During F-01 this produced a confident-but-false conclusion ("Zod Standard-Schema validators don't populate field errors") and a needless workaround + wrong comment â€” all because three "fixes" were tested against an unchanged server. A wrong test is worse than no test: it manufactures false certainty.
- **Rule**: Restart by PID/port, not name: `kill $(lsof -i :3000 -sTCP:LISTEN -t)`. Before trusting any test, confirm the **new** server bound (check the start log for `Ready` AND that the listening PID changed / no `EADDRINUSE`). When a fix "has no effect," suspect the harness before re-diagnosing the code.
- **Applies to**: /10x-implement, /10x-impl-review, any manual/E2E verification

## In Playwright, authenticate a supabase-js client with `signInWithPassword`, not browser-cookie reuse

- **Context**: Any E2E test that needs an **authenticated API client** (not just UI clicks) â€” e.g. seeding rows or asserting RLS isolation directly against PostgREST, where no mutation UI exists yet.
- **Problem**: Two traps that make the obvious approach fail. (1) `@supabase/ssr` stores the session as **HttpOnly, base64-chunked `sb-*-auth-token` cookies** â€” there is no JS-accessible token in the browser to "reuse from the signed-in context," so scraping `page.context().cookies()` is fiddly and coupled to ssr's encoding. (2) The Playwright **spec process loads no env** â€” `playwright.config.ts` only reads `process.env.CI`; the webServer gets `.env.local` because Next loads it natively, but a raw spec does not, and `@/lib/env` is a Next path alias unreachable from the test runner. So `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY` is undefined in the spec.
- **Rule**: Build the per-account client with `createClient(url, anonKey)` + `signInWithPassword(email, password)` using the test's own known credentials â€” it returns the `access_token` directly, no cookie extraction. Make the URL + anon key reachable to the spec by loading `.env.local` in `playwright.config.ts` (add `dotenv`, or inline the deterministic local anon key). Drive sign-up through the real UI for the auth-path coverage, but do **data ops** through the programmatic client.
- **Corollary**: Repo read helpers call `createClient()` (server, `next/headers` cookies) which throws outside a request â€” so any helper a test must call has to accept an **injectable** `SupabaseClient` (defaulting to `createClient()` in app code).
- **Applies to**: /10x-plan, /10x-implement, any Playwright E2E needing authenticated API access

## Verify Postgres constraints/schema via pg_catalog, not information_schema

- **Context**: Any time you probe Postgres schema during implement/review/research â€” confirming a FK's `ON DELETE` action, checking RLS policies, grants, or column definitions (e.g. proving `auth.sessions â†’ auth.users` cascades before relying on it).
- **Problem**: `information_schema`'s multi-view FK joins give false negatives for cross-schema FKs and only surface objects the current role has column privileges on. A constraint query returned **0 rows** for the real `auth.sessions â†’ auth.users ON DELETE CASCADE` that `pg_constraint` then confirmed â€” trusting it would have manufactured false certainty ("no cascade exists") and could have reverted a correct decision.
- **Rule**: When verifying constraints/FKs/schema, query the `pg_*` catalog (`pg_constraint`, `pg_class`, `pg_policy`, `pg_proc`) as the authoritative source â€” not `information_schema`. Treat an `information_schema` "absence" as unproven, not confirmed. (Same family as the verify-against-reality / pick-the-authoritative-probe rule.)
- **Applies to**: /10x-implement, /10x-impl-review, /10x-research, /10x-plan-review

## Local-Supabase E2E sign-up is intermittently flaky (GoTrue DB race) â€” don't gate on it

- **Context**: Running the Playwright suite (`pnpm test:e2e`) against the local Supabase stack. Every spec signs up a fresh user through the UI via the shared `e2e/helpers.ts` `signUp`.
- **Problem**: Single GoTrue container under sustained load over a long serial run. Two faces of the same cause: (1) sign-up intermittently fails to reach `/dashboard` and stays on `/sign-up` with an empty error alert ("Database error finding/saving user"); (2) a _later_ nav (e.g. `createNote` â†’ `toHaveURL(/notes/[id])`) times out because `proxy.ts` revalidates the session via GoTrue `GET /auth/v1/user` on every protected navigation, and those calls slow as the run accumulates (observed `/user` durations climbing 38â†’128ms). Not app code â€” it clears after a container restart. Red herrings: `[auth.rate_limit] email_sent` ("requires auth.email.smtp"; local uses Mailpit + `enable_confirmations = false` â†’ sign-up sends no email); `sign_in_sign_ups = 30/5min` is a real ceiling back-to-back reruns _can_ breach, but a 429 fails the sign-up POST (stuck-on-/sign-up), not a later nav.
- **Rule**: Environmental flake, don't gate on it. **Fixed permanently via `retries: 2` in `playwright.config.ts`** â€” `uniqueEmail()` is called inside each test body, so a retry reruns from the top with a brand-new account, side-stepping the race; `trace: 'on-first-retry'` still captures a real regression. Plus read-only specs (`dashboard.spec.ts`) reuse one shared session (a `setup` project saves `storageState`) to cut sign-up load â€” but mutation / empty-state / isolation specs **must** keep fresh-per-test sign-ups (they assert clean-slate / exact counts / distinct users; sharing a session swaps the flake for a contamination bug). For a stubborn local DB: `supabase db reset` (restarts containers â†’ clears GoTrue state).
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E on the local stack

## E2E must build a FRESH server â€” `reuseExistingServer: true` hijacks a running `next dev`

- **Context**: `pnpm test:e2e` while a `next dev` server is up on port 3000 (the normal case when developing one slice and testing another, or two parallel sessions). `playwright.config.ts` had `reuseExistingServer: !process.env.CI` (= `true` locally) with the webServer on the **same port 3000**.
- **Problem**: Playwright saw a server already on 3000 and **silently reused the `next dev` server** instead of running `pnpm build && pnpm start`. `next dev --turbo`'s on-demand compile + Fast Refresh is exactly the hydration race the prod-build was chosen to avoid: the sign-up form submits as a **native GET** (`/sign-up?email=â€¦&password=â€¦` in the URL) before React attaches its `onSubmit`, so it never POSTs to the Server Action â†’ **every sign-up-dependent spec fails deterministically** (~14/15). Misleading because it looks identical to the GoTrue flake above but is 100% reproducible, and a `supabase db reset` is a **red herring** (correlates only because a dev server happened to come up around the same time). Confirm the DB is innocent in one shot: `curl -X POST $URL/auth/v1/signup â€¦` returns a token â†’ the break is frontend, not the DB.
- **Rule**: E2E **always builds and tests a fresh production server, never a reused one** â€” `reuseExistingServer: false`. To let it coexist with a running `next dev`, isolate the e2e server on **both** axes: a dedicated **port (3100)** and a separate build dir (**`NEXT_DIST_DIR=.next-e2e`**, read by `next.config.ts`) so the e2e `next build` doesn't hit the dev server's `.next` build-lock ("Another next build process is already running"). Never flip `reuseExistingServer` back to `true` for speed â€” that is the bug. Diagnose a suspected hijack with `lsof -i :3000` + walking the parent chain to see `next dev` vs `next start`.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E run while a dev server may be up

## Parallel sessions share one working tree â€” stage by explicit path, never `git add -A`

- **Context**: Two Claude Code sessions working this repo at the same time (a normal mode here â€” e.g. S-08 in one, S-04/S-10/S-11 + a `getCurrentUser` cache refactor in another). They share **one on-disk working tree and one git index** â€” there is no per-session isolation unless a `git worktree` was created.
- **Problem**: The other session's half-finished edits sit dirty in the same tree. A `git add -A` / `git add .` (or any "stage everything" commit ritual) sweeps **their** uncommitted files into **your** commit â€” silently mixing unrelated work, polluting the slice's history, and risking shipping a half-built refactor. Same hazard for shared docs: editing `roadmap.md`/`CLAUDE.md` races the other session's edits â€” the Edit tool throws "file modified since read" mid-flight (observed twice closing S-08), and a blind overwrite would clobber their lines. The global `pnpm lint` also fails spuriously (`ENOENT .next-baseline/cache`) when the other session's build dir vanishes mid-glob.
- **Rule**: Stage **only the change's own paths, by name** (`git add <p1> <p2> â€¦`) â€” the `/10x-implement` ritual already prescribes this; honor it literally, never substitute `-A`/`.`. Detect concurrency early: `git reflog` showing commits you didn't author + a dirty tree at session start = another session is live. For shared docs, re-read immediately before each Edit and retry on "modified since read"; commit the doc edit on its own as soon as it lands so the window is tiny. Lint your slice's files explicitly (`pnpm exec eslint <files>`) to get a clean signal when the global run is poisoned by the other session's build dir. When work is genuinely independent and long-running, prefer a **`git worktree` per session** (as S-05 did) â€” it makes cross-contamination structurally impossible.
- **Applies to**: /10x-implement, /10x-archive, any commit/doc-edit step while a second session may be active

## Measure perf in a production build (`next start`), never in `next dev`

- **Context**: Diagnosing a slow page or deciding whether a perf problem is real â€” e.g. the `/subjects/[id]` document response reading **10.09s** in the browser Network tab, which drove a (premature) "this needs caching + a single-pane rewrite" conclusion.
- **Problem**: `next dev --turbo` is uncompiled and unoptimized â€” it compiles routes on demand and **re-runs server work uncached on every request** (RSC render + Shiki tokenization of every note body), so a heavy page's dev timing is routinely **3â€“10Ă— the production number** and is effectively meaningless for perf decisions. The 10s "emergency" was a dev-mode mirage: a real `pnpm build` + `next start` served the same subject in a fraction of the time (`Ready in 890ms`, sign-in 58ms). Trusting the dev figure would have escalated a UX-led experiment (single-pane subject view) into a fake perf fire and possibly justified a caching slice that wasn't needed yet.
- **Rule**: Before treating any latency as a bug or designing a fix, **reproduce it in a production build**. Coexist with a running `next dev` on :3000 by isolating on **both** axes (same trick as the E2E harness): a separate port (`next start -p 3200`) and a separate build dir (`NEXT_DIST_DIR=.next-prodtest pnpm build`, read by `next.config.ts`) so the build doesn't hit the dev `.next` lock. Read the prod number off the Network tab; only then decide. Same family as the verify-against-reality / authoritative-probe rule â€” a dev timing is "unproven," not "confirmed slow." (Independently, the Shiki highlighter loads **all ~200 bundled grammars** when `langs` is unconstrained â€” a measured **3.3s** one-time boot + ~129MB; constrain `langs` to the curated set + `lazy: true` â†’ ~140ms / ~22MB. That's a real, separate cold-start cost, but it is _not_ what the dev mirage was measuring.)
- **Applies to**: /10x-plan, /10x-implement, /10x-impl-review, /10x-research, any perf diagnosis

## URL-driven multiselect filter: debounce to batch, two-mode selection to avoid desync

- **Context**: A server-side list filter whose selection lives in the URL (`?subjects=a,b`) and re-queries on a Server Component on change (the notes "topic" filter; pattern lifted from `wykonczymy`'s `transfer-filters.tsx` + `FilterMultiSelect`). A client island toggles options; the page re-reads the param and re-runs the query.
- **Problem**: Three traps in the client island. (1) **Per-toggle round-trips** â€” committing the URL on every checkbox toggle fires one server re-query _per click_; picking 4 subjects = 4 re-queries + 4 list flashes. The fix is a **debounce that batches** a popover session into one trailing commit â€” this is a deliberate optimization, documented in the reference (`transfer-filters.tsx:66` "Debounce â€¦ batches rapid clicks to reduce how often we hit the server"), **not** incidental. (2) **Optimistic-state desync** â€” if the island seeds `useState` once from the URL prop, an external URL change (Back/Forward, manual edit) won't resync the checkmarks: `useState` ignores a changed initial value. (3) **Orphan timer** â€” a bare `setTimeout` with no cleanup can fire `router.replace` from an unmounted component.
- **Rule**: Use **two-mode selection**: while the popover is **open**, a local `useState` drives (instant toggles, debounce-batched); while **closed**, it's `null` and selection derives straight from the URL prop (`selected = localSelected ?? urlIds`) â€” so closed state always reflects the URL, killing the desync with no second source of truth. **Reseed** local from the prop on open; **flush** any pending debounce immediately on close; **clear** the timer in a cleanup-only `useEffect` (legit teardown, not the derived-state-in-effect anti-pattern the project rule bans). Wrap the commit in `useTransition` so the list stays interactive during the re-query. Keep the generic multiselect a pure controlled primitive (`components/ui/`, sibling to `combobox.tsx`); put the URL/debounce/two-mode logic in the feature-local filter wrapper. Expose **controlled `open`/`onOpenChange`** on the primitive so the wrapper can hook reseed/flush.
- **Meta-rule (why this lesson exists)**: A review flagged the timer-cleanup + desync gaps, and the first "fix" **deleted the debounce entirely** (swapped it for `useOptimistic` firing per-toggle) â€” trading a real, _documented_ optimization for a simpler-looking component. Wrong move: the reference already solved both gaps _with_ the debounce via the two-mode pattern. When review flags an **incomplete** implementation of a deliberate pattern, **complete the pattern** â€” don't strip the optimization. Before "simplifying" code mirrored from a reference, **read the reference's own comments for the rationale** (`rg -i debounce`), and treat a documented design decision as load-bearing until proven otherwise.
- **Applies to**: /10x-implement, /10x-impl-review, /simplify, any URL-state list filter
