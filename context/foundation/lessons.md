## Re-apply /10x-bootstrapper patches after every `10x get`

- **Context**: Any time a Module-1 lesson is re-fetched via `10x get` (m1l3/m1l4/m1l5…). The `.claude/skills/10x-bootstrapper/` files are vendored and overwritten with upstream's shipped versions on every fetch.
- **Problem**: Upstream still ships two bugs this repo patched — `audit_commands.js` as a bare `npm audit` string (fails `ENOLOCK` on pnpm) and a `.bootstrap-scaffold` temp-dir name (`create-next-app` rejects dot-prefixed names). A fetch silently reverts both, so a later `/10x-bootstrapper` run would break on this pnpm project.
- **Rule**: After any `10x get`, run `git checkout HEAD -- .claude/skills/10x-bootstrapper/` to re-apply the committed patches, then confirm `git status` shows those files unmodified.
- **Applies to**: all

## Verify against a server you confirmed bound — not a stale one

- **Context**: Manual/Playwright verification of a Next.js app via a local server. `next start` renames its process to `next-server`, so `pkill -f "next start"` never matches it; the old server keeps holding port 3000 and every "restart" silently fails with `EADDRINUSE`.
- **Problem**: The browser then hits the **stale build**, so code changes appear to have no effect. During F-01 this produced a confident-but-false conclusion ("Zod Standard-Schema validators don't populate field errors") and a needless workaround + wrong comment — all because three "fixes" were tested against an unchanged server. A wrong test is worse than no test: it manufactures false certainty.
- **Rule**: Restart by PID/port, not name: `kill $(lsof -i :3000 -sTCP:LISTEN -t)`. Before trusting any test, confirm the **new** server bound (check the start log for `Ready` AND that the listening PID changed / no `EADDRINUSE`). When a fix "has no effect," suspect the harness before re-diagnosing the code.
- **Applies to**: /10x-implement, /10x-impl-review, any manual/E2E verification

## In Playwright, authenticate a supabase-js client with `signInWithPassword`, not browser-cookie reuse

- **Context**: Any E2E test that needs an **authenticated API client** (not just UI clicks) — e.g. seeding rows or asserting RLS isolation directly against PostgREST, where no mutation UI exists yet.
- **Problem**: Two traps that make the obvious approach fail. (1) `@supabase/ssr` stores the session as **HttpOnly, base64-chunked `sb-*-auth-token` cookies** — there is no JS-accessible token in the browser to "reuse from the signed-in context," so scraping `page.context().cookies()` is fiddly and coupled to ssr's encoding. (2) The Playwright **spec process loads no env** — `playwright.config.ts` only reads `process.env.CI`; the webServer gets `.env.local` because Next loads it natively, but a raw spec does not, and `@/lib/env` is a Next path alias unreachable from the test runner. So `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY` is undefined in the spec.
- **Rule**: Build the per-account client with `createClient(url, anonKey)` + `signInWithPassword(email, password)` using the test's own known credentials — it returns the `access_token` directly, no cookie extraction. Make the URL + anon key reachable to the spec by loading `.env.local` in `playwright.config.ts` (add `dotenv`, or inline the deterministic local anon key). Drive sign-up through the real UI for the auth-path coverage, but do **data ops** through the programmatic client.
- **Corollary**: Repo read helpers call `createClient()` (server, `next/headers` cookies) which throws outside a request — so any helper a test must call has to accept an **injectable** `SupabaseClient` (defaulting to `createClient()` in app code).
- **Applies to**: /10x-plan, /10x-implement, any Playwright E2E needing authenticated API access

## Verify Postgres constraints/schema via pg_catalog, not information_schema

- **Context**: Any time you probe Postgres schema during implement/review/research — confirming a FK's `ON DELETE` action, checking RLS policies, grants, or column definitions (e.g. proving `auth.sessions → auth.users` cascades before relying on it).
- **Problem**: `information_schema`'s multi-view FK joins give false negatives for cross-schema FKs and only surface objects the current role has column privileges on. A constraint query returned **0 rows** for the real `auth.sessions → auth.users ON DELETE CASCADE` that `pg_constraint` then confirmed — trusting it would have manufactured false certainty ("no cascade exists") and could have reverted a correct decision.
- **Rule**: When verifying constraints/FKs/schema, query the `pg_*` catalog (`pg_constraint`, `pg_class`, `pg_policy`, `pg_proc`) as the authoritative source — not `information_schema`. Treat an `information_schema` "absence" as unproven, not confirmed. (Same family as the verify-against-reality / pick-the-authoritative-probe rule.)
- **Applies to**: /10x-implement, /10x-impl-review, /10x-research, /10x-plan-review

## Local-Supabase E2E sign-up is intermittently flaky (GoTrue DB race) — don't gate on it

- **Context**: Running the Playwright suite (`pnpm test:e2e`) against the local Supabase stack. Every spec signs up a fresh user through the UI via the shared `e2e/helpers.ts` `signUp`.
- **Problem**: Single GoTrue container under sustained load over a long serial run. Two faces of the same cause: (1) sign-up intermittently fails to reach `/dashboard` and stays on `/sign-up` with an empty error alert ("Database error finding/saving user"); (2) a _later_ nav (e.g. `createNote` → `toHaveURL(/notes/[id])`) times out because `proxy.ts` revalidates the session via GoTrue `GET /auth/v1/user` on every protected navigation, and those calls slow as the run accumulates (observed `/user` durations climbing 38→128ms). Not app code — it clears after a container restart. Red herrings: `[auth.rate_limit] email_sent` ("requires auth.email.smtp"; local uses Mailpit + `enable_confirmations = false` → sign-up sends no email); `sign_in_sign_ups = 30/5min` is a real ceiling back-to-back reruns _can_ breach, but a 429 fails the sign-up POST (stuck-on-/sign-up), not a later nav.
- **Rule**: Environmental flake, don't gate on it. **Fixed permanently via `retries: 2` in `playwright.config.ts`** — `uniqueEmail()` is called inside each test body, so a retry reruns from the top with a brand-new account, side-stepping the race; `trace: 'on-first-retry'` still captures a real regression. Plus read-only specs (`dashboard.spec.ts`) reuse one shared session (a `setup` project saves `storageState`) to cut sign-up load — but mutation / empty-state / isolation specs **must** keep fresh-per-test sign-ups (they assert clean-slate / exact counts / distinct users; sharing a session swaps the flake for a contamination bug). For a stubborn local DB: `supabase db reset` (restarts containers → clears GoTrue state).
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E on the local stack

## E2E must build a FRESH server — `reuseExistingServer: true` hijacks a running `next dev`

- **Context**: `pnpm test:e2e` while a `next dev` server is up on port 3000 (the normal case when developing one slice and testing another, or two parallel sessions). `playwright.config.ts` had `reuseExistingServer: !process.env.CI` (= `true` locally) with the webServer on the **same port 3000**.
- **Problem**: Playwright saw a server already on 3000 and **silently reused the `next dev` server** instead of running `pnpm build && pnpm start`. `next dev --turbo`'s on-demand compile + Fast Refresh is exactly the hydration race the prod-build was chosen to avoid: the sign-up form submits as a **native GET** (`/sign-up?email=…&password=…` in the URL) before React attaches its `onSubmit`, so it never POSTs to the Server Action → **every sign-up-dependent spec fails deterministically** (~14/15). Misleading because it looks identical to the GoTrue flake above but is 100% reproducible, and a `supabase db reset` is a **red herring** (correlates only because a dev server happened to come up around the same time). Confirm the DB is innocent in one shot: `curl -X POST $URL/auth/v1/signup …` returns a token → the break is frontend, not the DB.
- **Rule**: E2E **always builds and tests a fresh production server, never a reused one** — `reuseExistingServer: false`. To let it coexist with a running `next dev`, isolate the e2e server on **both** axes: a dedicated **port (3100)** and a separate build dir (**`NEXT_DIST_DIR=.next-e2e`**, read by `next.config.ts`) so the e2e `next build` doesn't hit the dev server's `.next` build-lock ("Another next build process is already running"). Never flip `reuseExistingServer` back to `true` for speed — that is the bug. Diagnose a suspected hijack with `lsof -i :3000` + walking the parent chain to see `next dev` vs `next start`.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E run while a dev server may be up

## Parallel sessions share one working tree — stage by explicit path, never `git add -A`

- **Context**: Two Claude Code sessions working this repo at the same time (a normal mode here — e.g. S-08 in one, S-04/S-10/S-11 + a `getCurrentUser` cache refactor in another). They share **one on-disk working tree and one git index** — there is no per-session isolation unless a `git worktree` was created.
- **Problem**: The other session's half-finished edits sit dirty in the same tree. A `git add -A` / `git add .` (or any "stage everything" commit ritual) sweeps **their** uncommitted files into **your** commit — silently mixing unrelated work, polluting the slice's history, and risking shipping a half-built refactor. Same hazard for shared docs: editing `roadmap.md`/`CLAUDE.md` races the other session's edits — the Edit tool throws "file modified since read" mid-flight (observed twice closing S-08), and a blind overwrite would clobber their lines. The global `pnpm lint` also fails spuriously (`ENOENT .next-baseline/cache`) when the other session's build dir vanishes mid-glob.
- **Rule**: Stage **only the change's own paths, by name** (`git add <p1> <p2> …`) — the `/10x-implement` ritual already prescribes this; honor it literally, never substitute `-A`/`.`. Detect concurrency early: `git reflog` showing commits you didn't author + a dirty tree at session start = another session is live. For shared docs, re-read immediately before each Edit and retry on "modified since read"; commit the doc edit on its own as soon as it lands so the window is tiny. Lint your slice's files explicitly (`pnpm exec eslint <files>`) to get a clean signal when the global run is poisoned by the other session's build dir. When work is genuinely independent and long-running, prefer a **`git worktree` per session** (as S-05 did) — it makes cross-contamination structurally impossible.
- **Applies to**: /10x-implement, /10x-archive, any commit/doc-edit step while a second session may be active

## Measure perf in a production build (`next start`), never in `next dev`

- **Context**: Diagnosing a slow page or deciding whether a perf problem is real — e.g. the `/subjects/[id]` document response reading **10.09s** in the browser Network tab, which drove a (premature) "this needs caching + a single-pane rewrite" conclusion.
- **Problem**: `next dev --turbo` is uncompiled and unoptimized — it compiles routes on demand and **re-runs server work uncached on every request** (RSC render + Shiki tokenization of every note body), so a heavy page's dev timing is routinely **3–10× the production number** and is effectively meaningless for perf decisions. The 10s "emergency" was a dev-mode mirage: a real `pnpm build` + `next start` served the same subject in a fraction of the time (`Ready in 890ms`, sign-in 58ms). Trusting the dev figure would have escalated a UX-led experiment (single-pane subject view) into a fake perf fire and possibly justified a caching slice that wasn't needed yet.
- **Rule**: Before treating any latency as a bug or designing a fix, **reproduce it in a production build**. Coexist with a running `next dev` on :3000 by isolating on **both** axes (same trick as the E2E harness): a separate port (`next start -p 3200`) and a separate build dir (`NEXT_DIST_DIR=.next-prodtest pnpm build`, read by `next.config.ts`) so the build doesn't hit the dev `.next` lock. Read the prod number off the Network tab; only then decide. Same family as the verify-against-reality / authoritative-probe rule — a dev timing is "unproven," not "confirmed slow." (Independently, the Shiki highlighter loads **all ~200 bundled grammars** when `langs` is unconstrained — a measured **3.3s** one-time boot + ~129MB; constrain `langs` to the curated set + `lazy: true` → ~140ms / ~22MB. That's a real, separate cold-start cost, but it is _not_ what the dev mirage was measuring.)
- **Applies to**: /10x-plan, /10x-implement, /10x-impl-review, /10x-research, any perf diagnosis

## URL-driven multiselect filter: debounce to batch, two-mode selection to avoid desync

- **Context**: A server-side list filter whose selection lives in the URL (`?subjects=a,b`) and re-queries on a Server Component on change (the notes "topic" filter; pattern lifted from `wykonczymy`'s `transfer-filters.tsx` + `FilterMultiSelect`). A client island toggles options; the page re-reads the param and re-runs the query.
- **Problem**: Three traps in the client island. (1) **Per-toggle round-trips** — committing the URL on every checkbox toggle fires one server re-query _per click_; picking 4 subjects = 4 re-queries + 4 list flashes. The fix is a **debounce that batches** a popover session into one trailing commit — this is a deliberate optimization, documented in the reference (`transfer-filters.tsx:66` "Debounce … batches rapid clicks to reduce how often we hit the server"), **not** incidental. (2) **Optimistic-state desync** — if the island seeds `useState` once from the URL prop, an external URL change (Back/Forward, manual edit) won't resync the checkmarks: `useState` ignores a changed initial value. (3) **Orphan timer** — a bare `setTimeout` with no cleanup can fire `router.replace` from an unmounted component.
- **Rule**: Use **two-mode selection**: while the popover is **open**, a local `useState` drives (instant toggles, debounce-batched); while **closed**, it's `null` and selection derives straight from the URL prop (`selected = localSelected ?? urlIds`) — so closed state always reflects the URL, killing the desync with no second source of truth. **Reseed** local from the prop on open; **flush** any pending debounce immediately on close; **clear** the timer in a cleanup-only `useEffect` (legit teardown, not the derived-state-in-effect anti-pattern the project rule bans). Wrap the commit in `useTransition` so the list stays interactive during the re-query. Keep the generic multiselect a pure controlled primitive (`components/ui/`, sibling to `combobox.tsx`); put the URL/debounce/two-mode logic in the feature-local filter wrapper. Expose **controlled `open`/`onOpenChange`** on the primitive so the wrapper can hook reseed/flush.
- **Meta-rule (why this lesson exists)**: A review flagged the timer-cleanup + desync gaps, and the first "fix" **deleted the debounce entirely** (swapped it for `useOptimistic` firing per-toggle) — trading a real, _documented_ optimization for a simpler-looking component. Wrong move: the reference already solved both gaps _with_ the debounce via the two-mode pattern. When review flags an **incomplete** implementation of a deliberate pattern, **complete the pattern** — don't strip the optimization. Before "simplifying" code mirrored from a reference, **read the reference's own comments for the rationale** (`rg -i debounce`), and treat a documented design decision as load-bearing until proven otherwise.
- **Applies to**: /10x-implement, /10x-impl-review, /simplify, any URL-state list filter

## Shiki `fallbackLanguage:'text'` emits `--shiki` vars on the `<pre>` only — scope highlight assertions to `span[style*="--shiki"]`

- **Context**: Asserting in E2E whether a markdown code block is syntax-highlighted vs plain — e.g. the S-13 guard that an unknown fence (` ```xyzzy `) degrades to plain text under `@shikijs/rehype`'s `lazy:true` + `fallbackLanguage:'text'`.
- **Problem**: A `text`-fallback block still renders as `<pre class="shiki" style="…--shiki-dark-bg:…;--shiki-dark:…">` — the dual-theme CSS vars sit on the **`<pre>` element**, while the inner token `<span>`s carry **no** style attr. So a "plain?" assertion written as "no `--shiki` anywhere in the `<pre>`" reads **2** and falsely fails even though the fallback worked. (Verified empirically with `codeToHtml('…', {lang:'text', themes})` before writing the assertion — count 2 on the `<pre>`, 0 on spans; a real `python` block puts `--shiki` on ~10 token spans.) Second trap: under `lazy:true` a valid _off-list_ language lazy-loads and highlights, so the fallback path only fires for a genuinely unresolvable token — the test must use a **bogus** language id, not a real off-list one.
- **Rule**: Distinguish highlighted from plain via **`pre.shiki span[style*="--shiki"]`** — `>3` for highlighted, `=== 0` for the fallback. Put the bogus-fence block in its **own note** so the count isn't contaminated by a highlighted block on the same page. Same family as verify-against-reality: render the real output and inspect it before asserting on its shape.
- **Applies to**: /10x-implement, /10x-impl-review, any E2E asserting Shiki highlight vs plain-text fallback

## A page in edit mode can render >1 CodeMirror — the shared `fillEditor` `.cm-content` locator goes ambiguous

- **Context**: E2E driving an inline-edit page that mounts more than one markdown/CodeMirror editor at once — e.g. S-14's `/notes/[id]?edit=note`, where `NoteForm` (the note body editor) renders **above** `MemoryCardsSection`'s always-present add/edit form (its `code_context` editor). The shared `e2e/helpers.ts` `fillEditor` targets a bare `.cm-content`.
- **Problem**: `fillEditor` was written for the single-editor case (`/notes/new` create: body editor only, since the inline-checks array starts empty). On a page with two editors, `page.locator('.cm-content')` resolves to **2 elements** → Playwright strict-mode violation, `locator.click` throws before any text is typed. The failure looks like a harness bug, but it's a true "which editor?" ambiguity the helper can't resolve.
- **Rule**: On any multi-editor page, don't call the bare shared `fillEditor` — scope to the intended instance: `page.locator('.cm-content').first()` (DOM order is reliable: the form rendered first in JSX is first), then `click()` + `evaluate(document.execCommand('insertText', …))` inline (same execCommand trick `fillEditor` uses, to dodge CodeMirror's closeBrackets corrupting ` ``` ` fences). Reserve the bare helper for genuinely single-editor pages (create/new). If multi-editor specs proliferate, add a `nth`/scoped variant to the helper rather than copy-pasting.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E on a page rendering multiple CodeMirror editors

## App-layer `z.uuid()` is stricter than Postgres `uuid` — validate DB ids with `z.guid()` (shape), not `z.uuid()` (RFC version/variant)

- **Context**: Validating an id (note / memory-card / subject) that originates in the DB and is echoed back through a Server Action — e.g. `/review`'s rating button sends `card.id` to `rateMemoryCard`, which parsed it with `memoryCardIdSchema = z.uuid()`. Surfaced as a red "Invalid memory card id" under the card, with every rating click reverting and no row written.
- **Problem**: Postgres `uuid` accepts ANY 128-bit value; Zod 4 `z.uuid()` enforces RFC-4122 (version nibble `1–8`, variant `8–b`). Deterministic seed ids like `…-0000-0000-…` (version 0) store fine in PG but `z.uuid()` rejects them, so every id-validating mutation fails at the schema and reverts silently. Doubly hidden: (a) Postgres AND TypeScript both consider the id well-formed, so nothing upstream complains; (b) `z.uuid()` **special-cases the nil uuid** (`00000000-0000-0000-0000-000000000000` passes) — so a unit test that used nil stayed green while every real version-0 id failed in the app. A prior session "fixed" it by crafting v4-shaped seed ids (commit `b63cf89`) — a DATA workaround that left the contract bug live for the next fixture / migration / externally-supplied id.
- **Rule**: For opaque, server-trusted ids that come from the DB, validate **shape only** with `z.guid()` — not `z.uuid()`. The DB owns id validity; the app's only job is to reject non-uuid-shaped junk (`z.guid()` still does that). Existence + ownership are enforced downstream by RLS + the action's re-fetch + the RPC's `if not found` — never by the version nibble. Reserve `z.uuid()` for ids you generate yourself and genuinely want to assert are RFC-v4. Lock the regression with a test that accepts a **non-nil version-0** id (nil won't catch it — see `src/__tests__/notes-schema.test.ts`). General principle (same family as verify-against-reality / authoritative-probe): validate an invariant at the layer that OWNS it; don't re-assert a stricter invariant the data was never required to satisfy.
- **Applies to**: /10x-implement, /10x-impl-review, /10x-plan, any Zod schema validating a DB-originated id

## dnd-kit `useSortable` spreads `role="button"` onto the row — `getByRole('listitem')` won't match it in E2E

- **Context**: E2E (Playwright) asserting on a list whose items are made draggable via `@dnd-kit` `useSortable` — e.g. the S-06 subject note-reorder list.
- **Problem**: `useSortable`'s `attributes` (spread onto the sortable element so it's keyboard-operable) include `role="button"` and `tabindex`. That overrides the element's implicit/`<li>` semantics, so `page.getByRole('listitem')` finds **0** rows even though they render — the test fails on a true accessibility-tree fact, not a bug.
- **Rule**: Don't select dnd-kit sortable rows by `listitem` role. Target a stable `data-testid` / text content instead, or query the role the widget actually exposes (`button`). If `listitem` semantics matter for a11y, render the drag affordance as a **handle** inside the `<li>` (spread `useSortable` attributes on the handle, not the row) — which S-15 plans anyway.
- **Applies to**: /10x-implement, /10x-impl-review, any Playwright E2E over a dnd-kit list

## E2E-server port must match Supabase `site_url` / email-template links

- **Context**: Any auth flow whose link is built server-side from a Supabase template — password recovery, email confirmation — exercised by E2E on the isolated e2e server (port 3100), while local Supabase's `site_url` points at the dev port (3000).
- **Problem**: `recovery.html` hardcoded `{{ .SiteURL }}` → the reset link pointed at `:3000`, but the e2e webServer runs on `:3100`, so the Playwright click landed on a dead port and the spec failed (S-06 en route, fixed `97cbf57`). The mismatch is invisible until a template-driven link is actually followed in a test.
- **Rule**: In email templates use **`{{ .RedirectTo }}`** (honors the per-request redirect the client passes) rather than the static `{{ .SiteURL }}`, so the link follows whichever origin initiated the flow (3000 dev, 3100 e2e). When adding any template-link auth flow, add an E2E that actually clicks the link, and confirm the port resolves.
- **Applies to**: /10x-implement, /10x-impl-review, any auth flow with a server-built email link tested in E2E

## After adding/moving/removing an App Router route, run `next typegen` before `pnpm typecheck`

- **Context**: Any slice that adds, moves, or deletes a `page.tsx`/`layout.tsx`/route segment under `src/app/` (Next 16 has typed routes on). S-15 added `/subjects/[id]/read/*`, then moved it up to `/subjects/[id]/*`.
- **Problem**: `pnpm typecheck` is bare `tsc --noEmit`, which reads Next's **generated** route types under `.next/dev/types/` (and `tsconfig` also globs `.next-e2e/types/**`, `.next-prodtest/types/**`). Those are stale until regenerated, so `tsc` fails with confusing errors against routes that are actually correct — `Type '"/subjects/[id]/read"' is not assignable to 'LayoutRoutes'`, or `Cannot find module '.../page.js'` for a route you just deleted. Two faces: (1) a NEW route isn't in the generated `LayoutRoutes`/`PageProps` union yet → false "not assignable"; (2) a stale build dir (`.next-prodtest` from the prod-build-server skill, `.next-e2e` from Playwright) still references a DELETED route's `page.js` → false "cannot find module".
- **Rule**: After any route-file add/move/delete, run **`pnpm exec next typegen`** before trusting `pnpm typecheck`. If the error points inside a stale `.next-prodtest`/`.next-e2e`, `rm -rf` that dir and re-run (both are rebuilt fresh on next use). Same family as verify-against-reality: a stale generated type is "unproven," not "broken code."
- **Applies to**: /10x-implement, /10x-impl-review, any slice touching `src/app/` routes

## App Router bracket segments (`[id]`, `[noteId]`) are git pathspec globs — stage with `:(literal)` or a parent dir

- **Context**: Any `git mv`/`git add`/`git rm` on files under bracketed route dirs — moving `subjects/[id]/read/*` up to `subjects/[id]/*`, staging a new `[noteId]/page.tsx`, etc.
- **Problem**: Git treats `[...]` in a pathspec as a **glob character class**, so `git add "src/app/(protected)/subjects/[id]/page.tsx"` matches a single-char dir named `i` or `d` — i.e. nothing — and the command either silently stages nothing or aborts the whole batch with `did not match any files` (one bad pathspec kills the rest, leaving everything unstaged). `git mv` mis-resolves the same way. Quoting in the shell does NOT help — it stops _bash_ globbing, but git applies its OWN pathspec glob after.
- **Rule**: For bracketed paths, either (a) prefix the pathspec with the literal magic `:(literal)` (e.g. `git add -- ':(literal)src/app/(protected)/subjects/[id]/page.tsx'`), or (b) stage a non-bracketed **parent directory** with `git add -A -- "src/app/(protected)/subjects"` when the whole subtree is yours (verify nothing foreign is dirty there first — parallel sessions share the tree). For deletions, `git add -A -- <path>` (plain `git add <deleted-path>` without `-A` can error). Prefer filesystem `mv` + `git add -A <parent>` over `git mv` for multi-file bracket moves.
- **Applies to**: /10x-implement, /10x-archive, any git file-op on App Router dynamic routes

## E2E selectors: `data-testid` for targeting, role/copy assertions only where the copy IS the contract

- **Context**: Choosing a Playwright locator for any element a spec interacts with. The suite leaned on `getByRole('button', { name: 'Delete' })`, `getByText('…')`, and `getByLabel('…')` — locators bound to user-visible copy + a11y wiring. Two recurring pains: (1) renaming a label / aria-text breaks specs that aren't even _about_ the copy (noise, not signal); (2) repeated elements (list cards, the dual desktop+mobile sidebar) match >1 node → strict-mode throws, patched with brittle `.first()` that silently grabs the wrong (hidden) node the day DOM order flips.
- **Problem**: Every selector couples to _something_; the failure mode is coupling to a thing that churns for reasons unrelated to the test. Copy churns constantly (rewording, i18n, tone); DOM structure churns on restyle; an `id` must be page-unique (useless for list rows) and gets co-opted for CSS/anchors. Coupling a locator to copy taxes _every_ refactor that touches wording, even when the test only needed "the submit button." For this solo MVP the team explicitly does **not** want specs failing on label changes.
- **Rule**: **Default to `getByTestId`** for locating elements; add `data-testid="<feature>-<element>"` (kebab-case: `subject-delete`, `auth-submit`, `note-title-input`) in the component. **Scope-then-target** for anything repeated: `getByTestId('subject-card').filter({ hasText })` → `.getByTestId('subject-delete')` — never `.first()`/`.nth()` to silence strict mode (positional `.nth()` is fine for a genuine "row 0 is A" assertion). **Decide per test** whether copy matters: where the wording IS the contract (auth CTA, a destructive-action label), assert it explicitly in the test body (`await expect(el).toHaveText('Create account')`) so a rename breaks _one_ clear assertion, not the locator — but only add that assertion when you actually care; by default, don't. - **Meta-rule**: a locator going stale on a label rename is noise; pay the copy-coupling tax once, deliberately, via an explicit `toHaveText` only where the copy is load-bearing. Targeting and verification are separate jobs — find by testid, verify by assertion.
- **Applies to**: /10x-implement, /10x-impl-review, /simplify, any Playwright E2E or component author choosing a selector

## `updated_at` is DB-owned via a `moddatetime` trigger — never hand-stamp it in app code

## Paginated PostgREST reads must handle the out-of-range 416 — `.range()` past the last row throws, and supabase-js drops the count

- **Context**: Any `{ rows, total }` list read built on `.select(cols, { count: 'exact' }).range(offset, offset + limit - 1)` (the getDueQueue precedent) — notes/memory-cards/subjects list queries. A deep-linked or stale `?page=N` whose `offset` lands past the last matching row.
- **Problem**: PostgREST returns **HTTP 416 / `PGRST103` "Requested range not satisfiable"** when the range start exceeds the row count — an **error**, not an empty result. So a query that treats any `error` as fatal (`if (error) throw`) **500s the whole page** on an out-of-range `?page=`. Worse, the Content-Range header still carries the real total (`*/131`) but **supabase-js does NOT parse `count` on the error path** (verified empirically: `count === null` on the 416), so you can't recover the total from the failed call to render a footer. The plan had _assumed_ out-of-range would yield "empty rows + a working footer" — it doesn't; it throws.
- **Rule**: Centralize paginated reads in a helper (`src/lib/supabase/run-paginated-query.ts`) that catches `error.code === 'PGRST103'` and falls back to a **lazy head-count** query (`.select(cols, { count: 'exact', head: true })` with the SAME filters, no range) to recover the total — returning `{ rows: [], total }` so the footer renders and the user can navigate back, instead of a 500. Build the filtered query via a `filtered(head: boolean)` factory so the ranged read and the count-only fallback share identical `.in()`/`.or()` predicates (the inner-join `notes!inner` count is correct by construction). One round-trip in the common in-range case; the second only fires on the rare overflow. Same family as verify-against-reality: probe the real 416 (`curl` + a supabase-js script) before assuming its shape.
- **Applies to**: /10x-implement, /10x-impl-review, /10x-plan, any `{rows,total}` paginated Supabase read

## Test-driven bugfixing — reproduce the bug with a failing test first

- **Context**: Any bug fix or debugging task, especially in mutation/persistence paths (Server Actions, RPCs, the recall loop). The inverse direction from risk-first test authoring.
- **Problem**: Fixing a bug without a reproducing test leaves no regression guard, and "fixed" gets asserted against the wrong signal — a swallowed error (try/catch that logs a warning but returns 200) lets the action report success while the real work failed, so a test on the response passes while the bug persists.
- **Rule**: Before fixing a bug, write a FAILING regression test that reproduces the symptom, pinned to the observable/persisted effect (the DB row/state column that was wrong), never the HTTP 200 or API response. It must fail before the fix and pass after, and stays in the repo as a permanent guard. Assert that status/state reflects the real failure, not that the call returned success. (Source: 10xDevs M3 L5.)

## Build-time env validation in Next.js needs a 3-file split + jiti — `server-only` can't reach `next.config`

- **Context**: Any Next.js env handling where `next build` must FAIL on a missing/invalid env var (client OR server) AND server secrets must stay out of the client bundle. Triggered when adding the first server-only var (e.g. SMTP creds) to a project whose `env.ts` is client-reachable.
- **Problem**: Two facts collide. `import 'server-only'` is the only NATIVE source of a **build-time** failure (a runtime Proxy à la t3-env, or a lazy check, only fails at runtime) — but `server-only` is Next-internal (no real `node_modules` package; aliased only during app compilation), so a module importing it CANNOT be imported by `next.config.ts` (plain Node), and it poisons the whole module for the client so it can't live in the client-reachable `env.ts`. Putting server vars in the eager client `env.ts` crashes the browser bundle; the references' single mixed `env.ts` "works" only by convention (never imported client-side) and fails only at runtime.
- **Rule**: Split env into three: a plain `env-schema.ts` (Zod `clientSchema`+`serverSchema`, no side effects, no `server-only`) importable by everything incl. `next.config`; `env.ts` (client, eager-parse `NEXT_PUBLIC_*` via STATIC `process.env.X` literals so Next inlines them); `env.server.ts` (`import 'server-only'` + eager parse — the build-time leak guard). Validate both at build start in `next.config.ts` via jiti: `createJiti(import.meta.url)`, then `jiti('./src/lib/env')` (client parse) + `serverSchema.parse(process.env)` (server, without importing the server-only module). jiti must be a direct devDep (pnpm won't resolve a transitive copy from next.config) and can't resolve the `@/` alias, so the env layer's internal imports use sibling relative paths. Reference impl: `src/lib/env*.ts` + `next.config.ts` (contact-form, archived `context/archive/2026-06-07-contact-form/`). Portable across other Next repos.
- **Applies to**: plan, implement, research

## A `z.union` request-body schema silently misroutes on fall-through — select the branch by raw discriminant presence, then validate

- **Context**: Any HTTP/Server-Action body validated by a `z.union([branchA, branchB])` whose members are NOT distinguishable by a literal discriminator (so `z.discriminatedUnion` doesn't apply) — e.g. `POST /api/memory-cards`, whose body is either `{ note_id, cards }` (attach) or `{ prompt, … , subject_id }` (standalone). The route picked the branch with `'note_id' in parsed.data` AFTER the union validated.
- **Problem**: Zod tries union members in order and accepts the first that passes. A body meant for branch A but malformed (`{ note_id, cards: 'garbage', prompt: '…' }`) FAILS branch A, then falls through to branch B, which **strips the unknown `note_id`** and validates the stray standalone fields — so a request that should 400 silently succeeds as the WRONG kind (a 201 standalone card instead of an error). Object schemas strip-by-default is what makes the fall-through invisible; typecheck/lint/build all stay green because it's a runtime routing gap, not a type error. Caught at the review gate (expose-cli-note-api, F1), not by the happy-path integration tests.
- **Rule**: When a request body has two shapes distinguished by the _presence_ of a key (not a literal tag), do NOT lean on a `z.union`'s fall-through to route it. Select the branch from the **raw** body first (`'note_id' in body`), then validate against that ONE schema so a malformed branch-A body 400s instead of being re-validated as branch B. Keep the union only for the inferred type, or drop it. Add a unit test asserting the malformed-discriminant input is REJECTED (not just that each happy path is accepted).
- **Applies to**: /10x-impl-review, /10x-implement, /10x-plan, /simplify, any Zod-validated multi-shape request body

## A Server Action that THROWS hangs the client transition in `isPending` — fail closed with a result

- **Context**: Any Server Action driven by `useActionTransition` (`hooks/use-action-transition.ts`) — the hook resolves its promise from the action's RETURNED `ActionResultT`. A helper the action calls that **throws** instead of returning (e.g. `isAccountEmpty` in `features/sample-data/queries.ts` re-throws a Supabase read error) breaks that contract. Surfaced in the sample-data wipe-then-load gate.
- **Problem**: When the action throws, `run`'s `await action()` rejects inside `startTransition`, so `resolve()` never fires — the promise never settles and `isPending` is stuck `true` forever (a spinner that never stops, no error toast, no inline error). ## `revalidatePath` on a dynamic (cookie-bound) page is a no-op — there's no cache to bust

- **Context**: Any Server Action that calls `revalidatePath('/some/route')` to "refresh" a page after a mutation, where that route is **dynamic** — it reads `cookies()` (every authed page here does, via the Supabase server client) and its data comes from **supabase-js** (not Next `fetch`/`unstable_cache`). Surfaced by the `revalidatePromptSurfaces` helper (deleted 2026-06-10): it revalidated four prompt-bearing routes after a Save/Reset so other surfaces would re-seed the editable system prompt.
- **Problem**: `revalidatePath` only does something if the route's output sits in a cache it can purge, and a dynamic route is in **none** of them: not the Full Route Cache (dynamic = not statically cached), not the Data Cache (supabase-js isn't Next `fetch`, so nothing is stored/tagged — which also makes `revalidateTag` a no-op), and not the client Router Cache (Next 15+ defaults `staleTimes.dynamic = 0`, so dynamic pages are refetched on every soft-nav, not reused). So the helper busted nothing — the surfaces were already re-reading the DB fresh on each navigation. It hides perfectly: typecheck/lint/build stay green (it's a valid call that just does nothing), and the feature "works" with or without it, so it reads as a load-bearing correctness mechanism. Verified both by the Next 16 `staleTimes` docs and a Playwright probe (out-of-band `user_prompts` change reflected after a soft-nav with no revalidate). Bonus tell: one of the four paths (`/memory-cards/new`) didn't even host the dialog it claimed to refresh.
- **Rule**: Before adding `revalidatePath`/`revalidateTag` to "refresh after a mutation", confirm the target route is actually **cached**. If it reads `cookies()` (dynamic) and its data is a supabase-js query (uncached), the call is dead code — the next navigation already re-renders fresh. Reach for revalidation only when you've deliberately cached something (`unstable_cache` + a tag, a static/ISR route); then `revalidateTag` the **data**, not a list of path strings. Don't keep a revalidate call as a "just in case" — it disguises dead code as a guarantee. (Residual it does NOT cover: browser bfcache, which `staleTimes` doesn't govern and `revalidatePath` doesn't reliably bust anyway.)
- **Applies to**: /10x-impl-review, /10x-implement, /10x-plan, /simplify, any Server Action that revalidates after a mutation
