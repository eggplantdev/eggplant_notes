# Rename "topic checks" → "memory cards" — Plan Brief

> Full plan: `context/changes/rename-topic-checks-to-memory-cards/plan.md`

## What & Why

Rename the recall-unit term `topic_checks` → `memory_cards` end-to-end. The current name fails on granularity ("topic" maps to nothing in the schema — the unit binds to a _note_) and the cleanup of UI/schema drift the owner wants closed before launch. "Memory card" won after rejecting "Card" (shadcn collision), "Review Card" (overloads the `review` session verb), and "Anki Card" (brand coupling + wrong model).

## Starting Point

The term spans ~70 files: 4 DB migrations (table, indexes, RLS, two RPCs, a `review_events` FK column), a generated `types.ts`, the `src/features/topic-checks/` feature dir, the `/topic-checks` route + nav, e2e + unit tests, and live foundation docs.

## Desired End State

No `topic[_-]?check` / `topicCheck` / `TopicCheck` anywhere in code, schema, routes, or live docs. `supabase db reset` applies clean, `types.ts` regenerates keyed on `memory_cards`, and the full suite is green. The app serves `/memory-cards` under a "Memory cards" nav label.

## Key Decisions Made

| Decision                       | Choice                      | Why                                                         | Source       |
| ------------------------------ | --------------------------- | ----------------------------------------------------------- | ------------ |
| Term                           | "memory card"               | Neutral, owns the model, no brand/UI collision              | Conversation |
| Scope                          | Schema-deep (not copy-only) | Copy-only leaves UI/schema drift the owner explicitly fears | Conversation |
| Migration strategy             | Rewrite migrations in place | Data disposable; `db reset` runs before launch              | Conversation |
| `review_events.topic_check_id` | Rename → `memory_card_id`   | Leaving it recreates the drift being killed                 | Plan         |
| `types.ts`                     | Regenerate, never hand-edit | It's generated by `pnpm db:types` → DB must change first    | Plan         |
| Old route redirect             | None                        | Solo MVP, no external links                                 | Plan         |

## Scope

**In scope:** DB schema/RLS/RPCs, `review_events.memory_card_id`, generated types, feature dir, route URL, nav, all identifiers + UI copy, e2e + unit tests, live `context/foundation/` docs.

**Out of scope:** `context/archive/` + `context/foundation/archive/` (immutable), the `review`/`review-events` vocabulary, the `review_events` and `notes` table names, the `create_note_with_checks` RPC name, data preservation, old-URL redirect.

## Architecture / Approach

Strictly ordered by the generated-types dependency: **DB → `db reset` + regenerate types → code/structure → docs → verify**. Deterministic casing map handles each variant (snake/kebab/Pascal/camel/prose).

- Overlap with the in-flight `review-on-dashboard` edits to `dashboard/page.tsx` etc. — must land after those commit.

- Full suite green: `typecheck`, `lint`, `test`, `build`, `test:e2e`.
- `/memory-cards` works; copy reads "memory card(s)" consistently.
