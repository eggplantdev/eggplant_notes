# Organize Notes into Subjects — Implementation Plan

## Overview

Add a grouping layer above the existing flat `notes`: a new `subjects` parent table and a
nullable, user-ordered `notes.subject_id` link. The user can create a subject (title +
optional description), assign notes to it from the note form, drag-reorder the notes within
a subject, and read a subject as one continuous top-to-bottom document — while each note
stays individually addressable and editable at its own route. Deleting a subject detaches
its notes (they survive as unassigned), never destroying them.

This is **S-06** (roadmap), Linear **EX-368**, PRD v2 **US-01** / Scope `[new] subjects` +
`[new] subject-as-document reading`. It is an additive structural change — no new domain
rule, no auth change, and (per PRD) no real data to preserve.

## Current State Analysis

The `notes` feature is a clean, single-pattern vertical that this slice extends 1:1:

- **Schema** (`supabase/migrations/20260603070945_init_notes_topic_checks_review_events.sql`):
  `notes(id, user_id not null references auth.users on delete cascade default auth.uid(),
title text, content text not null default '', created_at, updated_at)`. Per-action RLS uses
  the idiom `(select auth.uid()) = user_id` for select/insert/update/delete; btree index
  `notes_user_id_idx`. `topic_checks.note_id` FKs `notes(id) on delete cascade`.
- **Typed DB layer**: `src/lib/supabase/types.ts` is generated by `pnpm db:types`
  (`supabase gen types typescript --local`), committed alongside migrations. Both client
  factories (`src/lib/supabase/{server,client}.ts`) are parameterized `<Database>`.
- **Read/write wrappers**: `runTableQuery` (`src/lib/supabase/run-table-query.ts`, injectable
  client, **throws** on error/null) for reads; `runTableAction`
  (`src/lib/supabase/run-table-action.ts`, validates Zod, runs `.select().single()`, returns
  `TableActionResultT` discriminated union) for mutations. Actions **never send `user_id`** —
  the DB default + RLS `with check` own it.
- **notes feature** (`src/features/notes/`): `schemas.ts` (Zod `titleSchema`/`noteInputSchema`/
  `noteIdSchema`), `types.ts` (`NoteT = Database['public']['Tables']['notes']['Row']`),
  `queries.ts` (`getNotes`, `getNote` via `.maybeSingle()` → `undefined`), `actions/`
  (`createNote`/`updateNote`/`deleteNote`, each `revalidatePath` + `redirect`),
  `note-form.tsx` (TanStack `useAppForm`, union prop type for create vs edit),
  `delete-note-button.tsx` (AlertDialog).
- **Routes**: `(protected)/notes/page.tsx` (list), `/notes/[id]/page.tsx` (detail —
  `Promise.all([getNote, getTopicChecksForNote])`, `?edit` searchParam pattern),
  `/notes/new`, `/notes/[id]/edit`.

### Key Discoveries:

- **No ordering column exists anywhere** — `topic_checks` are immutably `created_at`-ordered
  (`src/features/topic-checks/queries.ts:44`). Ordering is greenfield; we choose the
  representation. **Decision: fractional `numeric` position** (drag-to-any-slot needs cheap
  single-row midpoint inserts, not an N-row renumber cascade).
- **No drag-and-drop library installed** (`package.json` confirmed). `@dnd-kit/core` +
  `@dnd-kit/sortable` get added in Phase 4.
- **shadcn footprint is minimal** — `src/components/ui/` has only `alert-dialog, button, card,
input, label, textarea`. **No `select`** — added in Phase 3 for the note-form subject picker.
- **PRD settles two of the three open questions**: a note may belong to one subject or none
  (`subject_id` nullable, no catch-all "Inbox"); no data to preserve, so the structural change
  is clean. The third — subject-delete behavior — is **decided here: detach (set null)**.

## Desired End State

A signed-in user can:

1. Create a subject with a title (+ optional description), rename it, and delete it.
2. Assign a note to a subject (or leave it unassigned) via a dropdown on the note create/edit
   form; unassigning is choosing "None".
3. Open `/subjects/[id]` and read every member note rendered in order as one continuous,
   Shiki-highlighted document, each section headed by the note title linking to that note's
   own editable page.
4. Drag-reorder the notes within a subject; the new order persists across reloads.
5. Delete a subject and find its former member notes intact and unassigned (cards/reviews
   untouched).
6. Never see or touch another user's subjects or note groupings (RLS enforced).

Verify: the Phase 5 Playwright suite is green, `pnpm typecheck/lint/build` pass, and the
recall loop + existing notes CRUD show no regression.

## What We're NOT Doing

- **No catch-all "Inbox" subject** — unassigned notes are free-floating (PRD AC).
- **No cascade-delete of notes** on subject delete — detach only.
- **No inline editing inside the document view** — sections link out to the note's own page.
- **No batch "add notes" picker on the subject page** — assignment is via the note form only
  (one note at a time).
- **No `'Add notes'` second assignment surface, no subject color/icon, no description rich
  text** — `description` is plain optional text.
- **No section-level (heading-anchor) navigation** — explicit PRD non-goal (v2-after).
- **No change to the recall loop, auth, or RLS idiom** — only additive policies.
- **No reverse "all cards from this note" view** — parked (roadmap).

## Implementation Approach

Mirror the `notes` vertical for `subjects` (schema → typed layer → queries/actions →
routes/UI), then thread a single nullable FK (`notes.subject_id`) + a `position` column
between them. Assignment reuses the existing `updateNote`/`createNote` path (extend the Zod
schema + insert/update payload). The drag-reorder is isolated to a **lightweight client island
that renders only note titles** (a reorderable table-of-contents) so the heavy Shiki document
body stays a server component; the island calls a dedicated `reorderNote` server action that
writes a single fractional `position`. Order everything so each phase is independently
verifiable: schema first, data layer second, static UI third, drag last (the static ordered
list from Phase 3 becomes draggable), E2E fifth.

## Critical Implementation Details

- **Fractional position semantics.** `position` is `numeric` (Postgres arbitrary precision —
  no float-precision cliff like `double`). On assign, `position = Date.now()` computed **in the
  action** — a fresh assign is always larger than existing members, so it appends to the end with
  **no read-before-write and no read-modify-write race** (this is the F2 plan-review fix; do NOT
  read `max(position)`). On reorder to a slot between neighbors `a` and `b`, the client computes
  `position = (a + b) / 2`; dropped at the start, `(0 + firstPos)/2`; at the end, `lastPos + 1`.
  Read order is `order by position asc nulls last, created_at asc` (tie-break stable). No
  `UNIQUE(subject_id, position)` constraint — midpoints are distinct in practice and a unique
  constraint would make a future rebalance harder; ties (should not occur) fall back to
  `created_at`. **`position` is null exactly when `subject_id` is null** — keep them
  consistent in every write path.
- **Reorder is one row.** The `reorderNote` action receives `(noteId, newPosition)` computed
  **client-side** from the visible neighbors (the island already holds the ordered list); the
  server validates ownership via RLS and writes only that row's `position`. Do not recompute
  the whole sequence server-side.
- **Assignment ownership (DB-enforced — F1 plan-review fix).** A user must not assign their note
  to another user's subject. The base `notes` UPDATE/INSERT policies only check
  `user_id = auth.uid()` on the note row — they do **not** validate that `subject_id` points to a
  subject the user owns (the FK guarantees existence, not ownership). Per this project's #1
  guardrail ("persistence-layer isolation enforced at the DB via RLS, not app code"), the Phase 1
  migration **extends the `notes` INSERT + UPDATE `with check`** to require
  `subject_id is null or exists (select 1 from subjects s where s.id = subject_id and s.user_id =
(select auth.uid()))`. The read-scoped picker (`getSubjects()`) is the UX layer; the RLS clause
  is the actual guarantee. The Phase 5 isolation E2E proves it (negative control: assigning to a
  foreign subject id is rejected).
- **Next.js 16**: `params`/`searchParams` are Promises in route components (see existing
  `/notes/[id]/page.tsx`); the drag island is `'use client'` and lazy where it would otherwise
  pull client weight into the document route.

## Phase 1: Migration + Typegen

### Overview

One new migration adds the `subjects` table with full per-action RLS and indexes, and alters
`notes` to add the nullable `subject_id` FK (`on delete set null`) and the `position numeric`
column. Regenerate the committed `Database` types.

### Changes Required:

#### 1. New migration

**File**: `supabase/migrations/<timestamp>_add_subjects_and_note_ordering.sql`

**Intent**: Create the `subjects` parent table mirroring the `notes` ownership/RLS pattern, and
extend `notes` with the grouping FK + ordering column. Additive only; no data backfill (no data
to preserve).

**Contract**:

- `subjects(id uuid pk default gen_random_uuid(), user_id uuid not null references auth.users(id)
on delete cascade default auth.uid(), title text not null, description text, created_at
timestamptz not null default now(), updated_at timestamptz not null default now())`.
- Four RLS policies on `subjects` for `authenticated` using the existing idiom
  `(select auth.uid()) = user_id` (select/insert with-check/update using+with-check/delete).
  `alter table subjects enable row level security;`
- Index `subjects_user_id_idx on subjects (user_id)`.
- `alter table notes add column subject_id uuid references subjects(id) on delete set null;`
  `alter table notes add column position numeric;`
- Index `notes_subject_id_position_idx on notes (subject_id, position);`
- **Ordering matters**: create `subjects` (table + policies) BEFORE altering `notes`, because the
  notes policy below references `subjects`.
- **Extend `notes` INSERT + UPDATE `with check` (F1 fix)** so a note can only point at a subject
  the caller owns. `alter policy "notes_update_own" on notes ... with check ((select auth.uid())
= user_id and (subject_id is null or exists (select 1 from subjects s where s.id =
notes.subject_id and s.user_id = (select auth.uid()))))`; same `with check` addition on
  `notes_insert_own`. (Use `alter policy`, or drop+recreate, preserving the existing
  `user_id` check.) SELECT/DELETE policies unchanged.

#### 2. Regenerate types

**File**: `src/lib/supabase/types.ts`

**Intent**: Reflect the new table and columns in the generated `Database` type so feature types
derive from it.

**Contract**: Run `pnpm db:types` after `supabase db reset` applies the migration. `notes.Row`
gains `subject_id: string | null` + `position: number | null`; a `subjects` table entry appears.

### Success Criteria:

#### Automated Verification:

- Migration applies cleanly: `supabase db reset` (local stack up) succeeds with no error
- Types regenerate and compile: `pnpm db:types` then `pnpm typecheck` passes
- RLS is enabled on `subjects`: `subjects` appears with `rowsecurity = true` in
  `pg_catalog.pg_tables` (per the lessons rule — verify via `pg_catalog`, not `information_schema`)
- Lint passes: `pnpm lint`

#### Manual Verification:

- In Supabase Studio, the `subjects` table and the two new `notes` columns are present
- Inserting a subject as the seed dev user works; selecting another user's subject returns nothing

**Implementation Note**: After automated verification passes, pause for manual confirmation
before Phase 2.

## Phase 2: Subjects Data Layer + Note Assignment

### Overview

Build the subjects feature module (schemas, types, queries, actions) mirroring `notes`, and
extend the note write path so assigning a subject also sets `position`.

### Changes Required:

#### 1. Subjects feature module

**File**: `src/features/subjects/schemas.ts`

**Intent**: Zod schemas for subject input + id.

**Contract**: `subjectInputSchema = { title: z.string().trim().min(1).max(200), description:
z.string().trim().max(2000).optional() }`; `subjectIdSchema = z.uuid()`. Export `SubjectInputT`.

**File**: `src/features/subjects/types.ts`

**Intent**: Derive the subject row type from the generated `Database`.

**Contract**: `SubjectT = Database['public']['Tables']['subjects']['Row']`.

**File**: `src/features/subjects/queries.ts`

**Intent**: RLS-scoped read helpers, injectable client (mirrors `notes/queries.ts`).

**Contract**:

- `getSubjects(client?)`: all subjects, `order by created_at desc` → `SubjectT[]` (via `runTableQuery`).
- `getSubject(id, client?)`: single via `.maybeSingle()` → `SubjectT | undefined` (mirrors `getNote`).
- `getNotesForSubject(subjectId, client?)`: notes where `subject_id = subjectId`,
  `order by position asc nulls last, created_at asc` → `NoteT[]` (via `runTableQuery`).
- `getUnassignedNotes(client?)`: notes where `subject_id is null` (for the "unassigned" home) → `NoteT[]`.

**File**: `src/features/subjects/actions/{create,update,delete}-subject.ts`

**Intent**: Server Actions mirroring the notes actions; never send `user_id`.

**Contract**: `createSubject(input)` inserts `{title, description}`, `revalidatePath('/subjects')`,
`redirect('/subjects/'+id)`. `updateSubject(id, input)` (rename + description) validates id,
updates, sets `updated_at`, revalidates list + detail. `deleteSubject(id)` deletes; FK `set null`
detaches member notes at the DB; revalidate `/subjects` + `/notes` (detached notes reappear in
list), redirect `/subjects`. All via `runTableAction`.

#### 2. Extend note assignment

**File**: `src/features/notes/schemas.ts`

**Intent**: Allow an optional `subject_id` on note input.

**Contract**: add `subject_id: z.uuid().nullable().optional()` to `noteInputSchema` ("None" in the
form maps to `null`).

**File**: `src/features/notes/actions/create-note.ts`, `update-note.ts`

**Intent**: Persist `subject_id` and keep `position` consistent with it.

**Contract**: When `subject_id` is non-null, set `position = Date.now()` (computed in the action
— append-to-end, **no `max()` read, no race**; F2 fix); insert/update `{title, content,
subject_id, position}`. When `subject_id` is null, set `position = null`. When updating and the
`subject_id` is unchanged, leave `position` as-is. Still never send `user_id`. (Cross-user
`subject_id` is rejected by the DB per the F1 RLS clause, not by the action.)

## Phase 4: Drag-to-Reorder

### Overview

Make the subject's note order user-editable via drag-and-drop, persisting a single fractional
`position` per move.

### Changes Required:

#### 1. dnd-kit dependency

**File**: `package.json` (+ `pnpm-lock.yaml`, `pnpm-workspace.yaml` if a build script surfaces)

**Intent**: Add the drag library.

**Contract**: `pnpm add @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities`. If pnpm reports
`ERR_PNPM_IGNORED_BUILDS`, add the offending package to `allowBuilds` in `pnpm-workspace.yaml`
(per AGENTS.md).

#### 2. Reorder server action

**File**: `src/features/subjects/actions/reorder-note.ts`

**Intent**: Persist one note's new fractional position.

**Contract**: `reorderNote(noteId, newPosition)` — validate `noteId` (uuid) and `newPosition`
(finite number) via Zod through `runTableAction`; `update notes set position = newPosition where
id = noteId` (RLS scopes to owner); `revalidatePath('/subjects/[id]')`. Returns
`TableActionResultT`. Server does **not** recompute the sequence — the client supplies the
midpoint.

#### 3. Sortable ToC island

**File**: `src/features/subjects/reorderable-note-list.tsx` (`'use client'`)

**Intent**: A lightweight draggable list of **note titles only** at the top of the subject page;
on drop, compute the midpoint position from the two visible neighbors and call `reorderNote` with
an optimistic reorder.

**Contract**: Props: ordered `{id, title, position}[]`. Uses `DndContext` + `SortableContext`
(`@dnd-kit`). `onDragEnd`: compute `newPosition` = midpoint of neighbor positions at the target
index (start = firstPos/2, end = lastPos+1), optimistically reorder local state, call
`reorderNote`; on failure, revert + surface an error. The heavy Shiki document body stays a
server component rendered below this island on `/subjects/[id]`.

## Performance Considerations

Negligible at MVP scale (tens of notes per subject). Fractional `numeric` positions make each
reorder a single-row update; the `(subject_id, position)` index keeps the ordered read cheap.
The document view renders Shiki server-side (no client highlight bytes), consistent with the
existing notes detail path.

**Deferred follow-up (F3, not this slice):** `/subjects/[id]` does N sequential async Shiki
highlight passes for N member notes (RenderMarkdown awaits Shiki per call). Fine at MVP scale,
but it is unbounded — the same class as the deferred S-02 finding ("unbounded per-note read +
Shiki-per-check"). Record it in `follow-ups/review-fixes.md` rather than addressing it here.

## Migration Notes

No data backfill — PRD confirms no real data to preserve. The migration is purely additive
(`subjects` table + two nullable `notes` columns); rollback is dropping them. Existing notes get
`subject_id = null`, `position = null` (valid unassigned state).
